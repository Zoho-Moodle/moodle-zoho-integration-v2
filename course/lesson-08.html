Lesson 08

Testing & Debugging

الاختبار والتصحيح

This lesson covers how to test the project using pytest, how to use the built-in debug endpoints, how to read error logs, and the most common errors you will encounter with their solutions.

يغطي هذا الدرس كيفية اختبار المشروع باستخدام pytest، كيف تستخدم نقاط نهاية التصحيح المدمجة، كيف تقرأ سجلات الأخطاء، وأكثر الأخطاء شيوعاً مع حلولها.

1. What is pytest? Why Write Tests?

١. ما هو pytest؟ لماذا نكتب اختبارات؟

pytest is a Python testing framework. A test is a small function that checks your code does what it is supposed to do. When you make changes to the project, running tests tells you instantly if anything broke.

pytest هو إطار عمل اختبار Python. الاختبار هو دالة صغيرة تتحقق من أن كودك يؤدي وظيفته. عندما تُجري تغييرات على المشروع، تشغيل الاختبارات يُخبرك فوراً إذا انكسر شيء.

# Simple example: testing the HMAC signature function
import hmac, hashlib
from app.core.security import verify_signature

def test_valid_signature():
    """A correct signature should be accepted."""
    secret  = "my-test-secret"
    payload = b'{"event":"test"}'
    sig     = hmac.new(secret.encode(), payload, hashlib.sha256).hexdigest()
    assert verify_signature(payload, sig, secret) == True

def test_invalid_signature():
    """A wrong signature should be rejected."""
    assert verify_signature(b'{"event":"test"}', "wrong-sig", "my-secret") == False

def test_tampered_payload():
    """Signature valid for one payload should not work for modified payload."""
    secret   = "my-secret"
    payload1 = b'{"event":"create"}'
    payload2 = b'{"event":"delete"}'   # attacker tries to swap the payload
    sig1     = hmac.new(secret.encode(), payload1, hashlib.sha256).hexdigest()
    assert verify_signature(payload2, sig1, secret) == False

كل دالة اختبار تبدأ بـ test_. تستخدم assert للتحقق من أن قيمة ما تساوي المتوقعة. إذا نجحت جميع الـ assert تنجح الاختبار، وإذا فشل أي منها تفشل الاختبار وترى رسالة تشرح ما ذهب خطأ.

2. Tests Folder Structure

٢. هيكل مجلد الاختبارات

backend/tests/
├── conftest.py           (shared fixtures — test DB, test client)
├── test_auth.py          (login, token validation tests)
├── test_student_sync.py  (sync logic tests)
├── test_webhooks.py      (HMAC verification, payload parsing tests)
├── test_mapper.py        (zoho_contact_to_student function tests)
└── test_db_models.py     (model creation, upsert pattern tests)

conftest.py — Shared Setup

The conftest.py file defines fixtures: shared setup code that any test can use. The most important fixture is a temporary test database so tests never touch your real moodle_zoho_local.db.

ملف conftest.py يُعرّف fixtures: كود إعداد مشترك يمكن لأي اختبار استخدامه. أهم fixture قاعدة بيانات مؤقتة للاختبار حتى لا تلمس الاختبارات قاعدة بياناتك الحقيقية.

import pytest
from sqlalchemy import create_engine
from sqlalchemy.orm import sessionmaker
from app.infra.db.base import Base

@pytest.fixture(scope="function")
def db():
    """Create a fresh in-memory SQLite database for each test function."""
    engine = create_engine("sqlite:///:memory:")  # memory only — deleted after test
    Base.metadata.create_all(bind=engine)         # create all tables
    Session = sessionmaker(bind=engine)
    session = Session()
    yield session   # hand the session to the test
    session.close()
    Base.metadata.drop_all(bind=engine)           # clean up tables

3. Running the Tests

٣. تشغيل الاختبارات

# Navigate to the backend folder
cd backend

# Activate your virtual environment first
.venv-2\Scripts\activate   # Windows
# or: source .venv-2/bin/activate   (Linux/Mac)

# Run all tests
pytest tests/ -v

# Run a specific test file
pytest tests/test_webhooks.py -v

# Run a specific test function
pytest tests/test_webhooks.py::test_valid_signature -v

# Show print() output during tests (useful for debugging)
pytest tests/ -v -s

# Stop on first failure
pytest tests/ -x

A passing test run looks like:

tests/test_mapper.py::test_zoho_to_student PASSED          [ 25%]
tests/test_webhooks.py::test_valid_signature PASSED        [ 50%]
tests/test_webhooks.py::test_invalid_signature PASSED      [ 75%]
tests/test_db_models.py::test_upsert_student PASSED        [100%]
========================= 4 passed in 0.23s ==========================

اختبار ناجح يظهر PASSED بالأخضر. اختبار فاشل يظهر FAILED باللحمر مع شرح ما ذهب خطأ. إذا لم تكن هناك اختبارات بعد يمكنك تشغيل pytest tests/ -v وستجد "no tests found".

4. The Debug API Endpoints

٤. نقاط نهاية API للتصحيح

The project has built-in debug endpoints to test individual components without running the full sync. Access them at http://localhost:8001/docs after starting the server.

المشروع يحتوي نقاط نهاية تصحيح مدمجة لاختبار المكونات الفردية دون تشغيل المزامنة الكاملة. يمكن الوصول إليها من http://localhost:8001/docs بعد تشغيل السيرفر.

EndpointWhat it doesUse for
GET /debug/zoho-connectionTests Zoho OAuth tokenVerify ZOHO_REFRESH_TOKEN works
GET /debug/moodle-connectionTests Moodle API tokenVerify MOODLE_TOKEN works
GET /debug/db-statsReturns row counts for all tablesQuick check how many records exist
POST /debug/sync-one-studentSyncs a single student by emailTest sync for one person without touching all data
POST /debug/test-webhookSends a test webhook to yourselfTest webhook processing without needing Zoho/Moodle

هذه النقاط موجودة في backend/app/routers/debug.py. يُفضَّل تعطيلها في بيئة الإنتاج (Production) بضبط DEBUG_ENDPOINTS_ENABLED=false في .env.

5. Reading the Admin Logs Page

٥. قراءة صفحة السجلات في لوحة التحكم

The easiest way to debug sync issues is to check the Admin Panel logs at http://localhost:8001/admin/logs. Every sync operation writes a record to the event_log table which is displayed here.

أسهل طريقة لتصحيح مشكلات المزامنة هي فحص سجلات لوحة التحكم على http://localhost:8001/admin/logs. كل عملية مزامنة تكتب سجلاً في جدول event_log يُعرَض هنا.

When you see an ERROR, the error_message column tells you exactly what went wrong. Common pattern: fix the root cause (e.g. update token in .env), restart the server, then trigger a manual sync to process the failed records.

عندما ترى خطأ، عمود error_message يُخبرك بالضبط ما ذهب خطأ. النمط الشائع: أصلح السبب الجذري (مثل تحديث الـ token في .env)، أعد تشغيل السيرفر، ثم شغّل مزامنة يدوية لمعالجة السجلات الفاشلة.

6. Common Errors and Solutions

٦. الأخطاء الشائعة وحلولها

ErrorCauseSolution
AUTHENTICATION_FAILURE (Zoho)Refresh token expired/revokedGenerate a new refresh token from api-console.zoho.eu → update .env → restart
401 Unauthorized (Moodle)Wrong or expired MOODLE_TOKENCreate a new token in Moodle admin → update MOODLE_TOKEN in .env → restart
UNIQUE constraint failed: students.emailDuplicate email in Zoho dataClean up duplicate contacts in Zoho CRM; our upsert uses zoho_id not email as primary key — check for contacts with same email but different IDs
Connection refused: 127.0.0.1:8001Server is not runningcd backend and run python start_server.py
ModuleNotFoundError: No module named 'fastapi'Wrong Python env activeRun: .venv-2\Scripts\activate then try again
No .env file foundMissing .env configurationCopy .env.example to .env and fill in all required values
Moodle API error: invalidtokenMoodle token wrong or expiredSite Admin → Server → Web services → Manage tokens → regenerate
When the server prints a traceback (stack trace), READ it from the bottom up. The last line is the actual error. Lines above it show what code path led to the error. Copy the last 3-5 lines into a search engine — someone has solved it before.

عندما يطبع السيرفر traceback (مسار الخطأ)، اقرأه من الأسفل للأعلى. السطر الأخير هو الخطأ الفعلي. انسخ آخر 3-5 أسطر في محرك البحث — شخص ما حلّه من قبل.
