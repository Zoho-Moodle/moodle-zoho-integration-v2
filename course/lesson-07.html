Lesson 07

Webhooks Architecture

هندسة الـ Webhooks

This lesson explains what a webhook is, why it is better than polling, how HMAC-SHA256 signature verification works, and a full walkthrough of how Zoho and Moodle webhooks are processed by the backend.

يشرح هذا الدرس ما هو الـ webhook، لماذا هو أفضل من الاستطلاع الدوري، كيف يعمل التحقق من التوقيع HMAC-SHA256، واستعراض كامل لكيفية معالجة الـ webhooks من Zoho و Moodle.

1. Push vs Pull — Why Webhooks Exist

١. الدفع مقابل السحب — لماذا الـ Webhooks؟

There are two ways for our backend to know when something changes in Zoho or Moodle:

هناك طريقتان لمعرفة الباك-إند بأن شيئاً تغيّر في Zoho أو Moodle:

PULL (Polling) — like constantly checking your phone for messages:
  Our server → asks Zoho "any new students?" → Zoho replies
  Our server → asks Zoho "any new students?" → Zoho replies "no"
  Our server → asks Zoho "any new students?" → Zoho replies "no"
  (repeat every 5 minutes, even when nothing changed)
  Cons: wastes API quota, wastes bandwidth, not instant (5 min delay)

PUSH (Webhook) — like receiving a notification when a message arrives:
  Zoho detects new student is saved → Zoho sends POST request to our URL
  Our server receives it → processes it INSTANTLY
  Pros: real-time, zero wasted API calls when nothing changes

This project uses webhooks for real-time events (new student saved, grade updated) and scheduled pull sync as a safety net (nightly full sync catches anything missed).

هذا المشروع يستخدم webhooks للأحداث الفورية (طالب جديد، تحديث درجة) ومزامنة مجدوَلة كشبكة أمان (مزامنة ليلية كاملة لالتقاط أي شيء فات).

2. What is HMAC-SHA256? Verifying the Sender

٢. ما هو HMAC-SHA256؟ التحقق من المُرسِل

When Zoho (or our Moodle plugin) sends a webhook to our server, anyone on the internet could potentially send a fake request to the same URL. HMAC-SHA256 is a cryptographic signature that proves the request is genuinely from Zoho/Moodle and not from an attacker.

عندما يُرسل Zoho أو إضافة Moodle webhook لسيرفرنا، يستطيع أي شخص على الإنترنت إرسال طلب مزيّف لنفس الرابط. HMAC-SHA256 هو توقيع تشفيري يُثبت أن الطلب حقيقي من Zoho/Moodle وليس من مهاجم.

HOW HMAC SIGNATURE WORKS:
=========================
Sender side (Zoho or Moodle plugin):
  payload  = '{"event":"student_saved","id":"123"}'
  secret   = "our-shared-secret-from-.env"
  signature = HMAC-SHA256(payload, secret)
  → sends: HTTP POST with header "X-Zoho-Signature: abc123def456..."

Receiver side (our FastAPI backend):
  received_body = request.body()
  secret = settings.WEBHOOK_SECRET  (same secret from .env)
  expected_signature = HMAC-SHA256(received_body, secret)
  if received_signature == expected_signature:
    ✅ VALID — process the webhook
  else:
    ❌ REJECT — return 401 Unauthorized (possible attack)

The magic: HMAC can only be computed by someone who knows the SECRET.
Zoho knows it (you configured it there). We know it (.env).
An attacker does NOT know it → cannot forge the signature.
    
import hmac, hashlib

def verify_signature(body: bytes, received_sig: str, secret: str) -> bool:
    """
    Verify that the webhook request came from a trusted sender.
    body         = raw request bytes (MUST be bytes, not string!)
    received_sig = value of the X-Zoho-Signature (or X-Moodle-Signature) header
    secret       = WEBHOOK_SECRET from .env
    """
    expected = hmac.new(
        key    = secret.encode("utf-8"),
        msg    = body,
        digestmod = hashlib.sha256
    ).hexdigest()

    # Use hmac.compare_digest — prevents timing attacks
    # (a plain == comparison could leak info via timing differences)
    return hmac.compare_digest(expected, received_sig)

3. The Zoho Webhook Endpoint

٣. نقطة نهاية Zoho Webhook

Zoho sends a POST to /webhooks/zoho when a Contact is created or updated. Here is how the endpoint works in backend/app/routers/webhooks.py:

Zoho يُرسل POST إلى /webhooks/zoho عند إنشاء أو تحديث جهة اتصال. إليك كيف تعمل نقطة النهاية في backend/app/routers/webhooks.py:

from fastapi import APIRouter, Request, HTTPException, Depends
from sqlalchemy.orm import Session
from app.infra.db.session import get_db
from app.core.config import settings
from app.integrations.zoho.mapper import zoho_contact_to_student
import json

router = APIRouter(prefix="/webhooks", tags=["Webhooks"])

@router.post("/zoho")
async def zoho_webhook(request: Request, db: Session = Depends(get_db)):
    # Step 1: Read raw body bytes BEFORE parsing JSON
    # (must use raw bytes for HMAC — JSON parsing would lose byte-exact content)
    body = await request.body()

    # Step 2: Verify the HMAC signature
    received_sig = request.headers.get("X-Zoho-Signature", "")
    if not verify_signature(body, received_sig, settings.WEBHOOK_SECRET):
        raise HTTPException(status_code=401, detail="Invalid signature")

    # Step 3: Parse JSON payload
    payload = json.loads(body)
    event_type = payload.get("event")
    contact    = payload.get("data", {})

    # Step 4: Process based on event type
    if event_type in ("contact_created", "contact_updated"):
        student_data = zoho_contact_to_student(contact)
        student = upsert_student(db, student_data)
        log_event(db, "webhook_zoho", "success", entity_id=student.zoho_id)
        return {"status": "processed", "student_id": student.id}

    # Unknown event type — acknowledge but don't process (prevents Zoho retries)
    return {"status": "ignored", "event": event_type}
Always return HTTP 200 quickly! If your server takes more than 10 seconds, Zoho considers the delivery failed and retries. Lengthy operations should be dispatched to a background task with BackgroundTasks and the endpoint should return immediately.

دائماً أرجع HTTP 200 بسرعة! إذا تأخر السيرفر أكثر من 10 ثوانٍ، يعتبر Zoho أن الإرسال فشل ويُعيد المحاولة. العمليات الطويلة يجب تشغيلها كـ background task.

4. Setting Up the 16 Zoho Workflow Rules — Auto-Setup

٤. إعداد الـ 16 Workflow Rule في Zoho — الإعداد التلقائي

The project uses 8 custom BTEC modules, each needing 2 Workflow Rules (upsert + delete) = 16 rules total. Instead of creating them manually in the Zoho UI, the backend does it automatically:

يستخدم المشروع 8 وحدات BTEC مخصصة، كل وحدة تحتاج Workflow Rule اثنين (upsert + delete) = 16 rule إجمالاً. بدلاً من إنشائها يدوياً في واجهة Zoho، ينشئها الباك-إند تلقائياً:

  1. Make sure WEBHOOK_BASE_URL in .env points to your public server URL (use ngrok for local dev: ngrok http 8001)
  2. Call POST /api/v1/webhooks/setup from Swagger or curl
  3. The backend's ZohoWorkflowService in zoho_workflow_service.py:
    • Creates 16 Zoho Webhook entities (one per module/trigger combination)
    • Creates 16 Workflow Rules that reference those webhooks by ID
    • Saves all IDs to zoho_rules_state.json for cleanup/re-setup

5. Idempotency — Safe to Receive the Same Webhook Twice

٥. الأمان من التكرار — استقبال نفس الـ Webhook مرتين بأمان

Webhooks are not guaranteed to be delivered exactly once. Network issues can cause Zoho to retry a webhook that was already processed. Idempotency means processing the same event multiple times produces the same result — no duplicate records.

وصول الـ webhook مضمون ليس تماماً مرة واحدة. مشاكل الشبكة تجعل Zoho يُعيد إرسال webhook تمت معالجته بالفعل. الأمان من التكرار يعني أن معالجة نفس الحدث مرات متعددة تُنتج نفس النتيجة — بدون سجلات مكررة.

The upsert pattern (from Lesson 03) is what makes our webhooks idempotent. Because we always do "find existing record by zoho_id → update if found, create if not found", processing the same BTEC record twice just runs two identical updates — never creates a duplicate.

نمط الـ upsert (من الدرس 03) هو ما يجعل webhooks آمنة من التكرار. لأننا دائماً نبحث بـ zoho_id ونُحدّث إن وُجد وننشئ إن لم يوجد، معالجة نفس سجل BTEC مرتين تُجري تحديثين متطابقين — لا تُنشئ نسخة مكررة أبداً.

Best practice: also store a webhook_event_id (unique ID that Zoho includes in each webhook payload) in your event_log table. If you see the same event_id twice, skip processing immediately.

أفضل ممارسة: خزّن الـ webhook_event_id (المعرّف الفريد الذي يُضمّنه Zoho في كل payload) في جدول event_log. إذا رأيت نفس المعرّف مرتين، تخطَّ المعالجة فوراً.

6. The Moodle Plugin Webhook Flow

٦. تدفق Webhook من إضافة Moodle

The Moodle plugin sends webhooks to /webhooks/moodle. The flow is similar to the Zoho webhook but the trigger is Moodle events (enrollment, grade update).

إضافة Moodle ترسل webhooks إلى /webhooks/moodle. التدفق مشابه لـ webhook Zoho لكن المُحرِّك هو أحداث Moodle (التسجيل، تحديث الدرجة).

MOODLE WEBHOOK FLOW:
====================
1. Student gets enrolled in a Moodle course
2. Moodle fires event: user_enrolment_created
3. Our plugin's observer.php catches this
4. observer.php computes HMAC signature using shared secret
5. observer.php sends POST to: {BACKEND_URL}/webhooks/moodle
6. FastAPI /webhooks/moodle endpoint:
   a. Verifies HMAC signature
   b. Reads event type + user_id + course_id
   c. Finds student in local DB by moodle_id
   d. Creates/updates Enrollment record in local DB
   e. Returns {"status": "ok"} so Moodle does not retry
