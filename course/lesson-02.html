02

FastAPI Backend & Admin Panel

Understand how the FastAPI application is assembled, how requests flow through it, and how the Admin Panel is served and secured.

الباك اند ولوحة التحكم — كيف يعمل كل شيء من الداخل

1 — What Is FastAPI?

١ — ما هي FastAPI؟

FastAPI is a modern Python web framework for building REST APIs. A web framework is a library that handles the low-level HTTP plumbing so you can focus on application logic. Without a framework, you would have to manually parse HTTP headers, read request bodies, handle routing, and format JSON responses — hundreds of lines of boilerplate. FastAPI handles all of that and gives you a clean, declarative API to work with.

Think of FastAPI as a waiter in a restaurant. You (the developer) write the menu (the endpoints and what they do). FastAPI is the waiter who:

  • Takes the customer's order (receives the HTTP request).
  • Validates it — "did the customer ask for something that's on the menu? Are all required fields present?" (Pydantic validation).
  • Hands the order to the kitchen (calls your handler function).
  • Brings back the meal in the right format (serializes your Python object to JSON and returns the HTTP response).

Why FastAPI over other frameworks (like Flask or Django)?

  • Speed: FastAPI is built on Starlette (an async ASGI framework) and is one of the fastest Python web frameworks — comparable to NodeJS and Go in throughput.
  • Automatic documentation: Just by defining your endpoint and its type hints, FastAPI generates a complete, interactive Swagger UI at /docs — no extra work needed.
  • Type safety: FastAPI uses Python type hints everywhere. If a request sends a string where you declared an int, FastAPI rejects it with a clear error before your handler function is even called.
  • Dependency injection: FastAPI has a clean built-in system for declaring shared dependencies (database sessions, authentication) that gets injected into handler functions automatically.

FastAPI هو إطار ويب Python حديث لبناء REST APIs. إطار الويب هو مكتبة تتعامل مع الأجزاء الدنيا من HTTP حتى تتفرّغ لمنطق تطبيقك.

فكّر في FastAPI كـ نادل في مطعم. أنت (المطوّر) تكتب القائمة (نقاط النهاية وما تفعله). FastAPI هو النادل الذي: يستقبل طلب الزبون (HTTP request)، يتحقق من صحته (Pydantic validation)، يمرّره للمطبخ (يستدعي دالة المعالج)، ويُعيد الطبق بالشكل الصحيح (JSON response).

لماذا FastAPI؟ السرعة: من أسرع أطر Python. التوثيق التلقائي: بمجرد تعريف نقاط النهاية يُولّد FastAPI Swagger UI تفاعلياً على /docs تلقائياً. أمان الأنواع: يستخدم type hints ويرفض طلبات غير صحيحة قبل استدعاء المعالج. حقن التبعيات: نظام نظيف لإعلان التبعيات المشتركة (جلسة DB، المصادقة) تُحقن تلقائياً.

2 — How app/main.py Works

٢ — كيف يعمل app/main.py

app/main.py is the entry point of the FastAPI application. When uvicorn starts with "app.main:app", it imports this file and retrieves the object named app — a FastAPI instance. Let's walk through what happens in this file step by step.

Step 1: Create the FastAPI application instance

from fastapi import FastAPI
from app.core.config import settings
from contextlib import asynccontextmanager

@asynccontextmanager
async def lifespan(app: FastAPI):
    # ── STARTUP ──
    from app.infra.db.base import Base, engine
    Base.metadata.create_all(bind=engine)  # create DB tables if missing
    from app.core.scheduler import start_scheduler
    start_scheduler()                       # begin background sync jobs
    yield
    # ── SHUTDOWN ──
    from app.core.scheduler import stop_scheduler
    stop_scheduler()                        # graceful shutdown

app = FastAPI(
    title=settings.APP_NAME,
    version="3.0.0",
    lifespan=lifespan
)

The lifespan context manager is a FastAPI feature (introduced in 0.93+) that replaces the older @app.on_event("startup") pattern. Code before the yield runs on startup; code after yield runs on shutdown. This is where we create database tables and start the scheduler.

Step 2: Add CORS middleware

from fastapi.middleware.cors import CORSMiddleware

app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],          # or list specific Moodle domains
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)

What is CORS middleware? When a browser page on moodle.school.com makes an AJAX request to api.school.com:8001, the browser first sends a "preflight" OPTIONS request to ask: "is this server OK with requests from this origin?" The CORS middleware answers this preflight with the appropriate Access-Control-Allow-Origin headers, telling the browser it is safe to proceed. Without this middleware, all cross-origin calls from Moodle's frontend would be blocked by the browser — even if the server physically received them.

Step 3: Mount the API and Admin routers

from app.api.v1.router import api_router
from app.admin.router import admin_router
from fastapi.staticfiles import StaticFiles

app.include_router(api_router, prefix="/api/v1")
app.include_router(admin_router)

# Serve CSS/JS for the admin panel
app.mount("/static", StaticFiles(directory="admin/static"), name="static")

app.include_router(api_router, prefix="/api/v1") takes all routes defined in api_router and makes them available under the /api/v1/ prefix. If a route is defined as @router.post("/sync/students") inside endpoints/sync_students.py, it becomes POST /api/v1/sync/students in the running application.

Step 4: Add the health check

@app.get("/health")
async def health():
    return {"status": "ok", "app": settings.APP_NAME}

app/main.py هو نقطة دخول تطبيق FastAPI. عند بدء uvicorn بـ "app.main:app"، يستورد هذا الملف ويجلب الكائن app.

دورة الحياة (lifespan): الكود قبل yield يعمل عند بدء التشغيل (إنشاء جداول DB، تشغيل المجدوّل)؛ الكود بعد yield يعمل عند الإيقاف (إيقاف المجدوّل).

CORS middleware: عند محاولة صفحة متصفح على moodle.school.com إرسال طلب AJAX إلى api.school.com:8001، يرسل المتصفح أولاً طلب preflight OPTIONS. CORS middleware يرد بالترويسات المناسبة للسماح بالطلب. بدون هذا، تُحجب جميع الطلبات cross-origin من واجهة Moodle.

app.include_router(api_router, prefix="/api/v1"): يأخذ جميع المسارات المعرّفة في api_router ويجعلها متاحة تحت بادئة /api/v1/.

3 — What Is a Router?

٣ — ما هو الراوتر (Router)؟

In FastAPI, a router (APIRouter) is like a mini-application that can hold a group of routes. You define routes on the router inside a feature module, then include that router in the main app. This is exactly like chapters in a book — the main app is the book, and each router is a chapter with its own set of endpoints.

How it works in practice:

# File: app/api/v1/endpoints/sync_students.py
from fastapi import APIRouter, Depends
from sqlalchemy.orm import Session
from app.infra.db.session import get_db
from app.services.sync import sync_students_service

router = APIRouter()

@router.post("/students", summary="Sync all students from Moodle")
async def sync_students_endpoint(db: Session = Depends(get_db)):
    result = sync_students_service(db)
    return result
# File: app/api/v1/router.py
from fastapi import APIRouter
from .endpoints import sync_students, sync_grades, webhooks, mappings

api_router = APIRouter()

api_router.include_router(
    sync_students.router,
    prefix="/sync",
    tags=["Sync"]
)
api_router.include_router(
    sync_grades.router,
    prefix="/sync",
    tags=["Sync"]
)
api_router.include_router(
    webhooks.router,
    prefix="/webhooks",
    tags=["Webhooks"]
)
api_router.include_router(
    mappings.router,
    prefix="/mappings",
    tags=["Mappings"]
)
# File: app/main.py (excerpt)
from app.api.v1.router import api_router
app.include_router(api_router, prefix="/api/v1")

The final URL for the student sync endpoint is assembled from three pieces: /api/v1 (from main.py) + /sync (from router.py) + /students (from the endpoint decorator) = /api/v1/sync/students.

The tags=["Sync"] parameter is used by Swagger UI to group related endpoints under a collapsible section. All sync-related endpoints appear together under a "Sync" heading in the /docs page, making the documentation much easier to navigate.

في FastAPI، الراوتر (APIRouter) هو تطبيق مصغّر يحمل مجموعة من المسارات. تعرّف المسارات على الراوتر داخل وحدة الميزة، ثم تضم ذلك الراوتر في التطبيق الرئيسي. تماماً كالفصول في كتاب — التطبيق app هو الكتاب، وكل راوتر فصل بمجموعة نقاط نهايته.

URL النهائي لنقطة نهاية مزامنة الطلاب يُجمّع من ثلاثة أجزاء: /api/v1 (من main.py) + /sync (من router.py) + /students (من ديكوراتور النقطة) = /api/v1/sync/students.

معامل tags=["Sync"] يستخدمه Swagger UI لتجميع نقاط النهاية ذات الصلة تحت قسم قابل للطيّ في صفحة /docs، مما يجعل التوثيق أسهل في التنقّل.

4 — Understanding a Single Endpoint Step by Step

٤ — فهم نقطة نهاية واحدة خطوة بخطوة

Let's dissect a complete, realistic endpoint and understand every line. We will use the student sync endpoint: 

from fastapi import APIRouter, Depends, HTTPException, status
from sqlalchemy.orm import Session
from typing import Dict, Any

from app.infra.db.session import get_db
from app.services.sync import sync_students_service
from app.admin.auth import require_any_role

router = APIRouter()

@router.post(
    "/students",
    summary="Pull students from Moodle and upsert locally",
    response_description="Sync summary with counts",
    status_code=status.HTTP_200_OK
)
async def sync_students(
    db: Session = Depends(get_db),
    _user = Depends(require_any_role)
) -> Dict[str, Any]:
    """
    Triggers a full pull of all user accounts from Moodle's
    core_user_get_users web service, then upserts each user
    into the local students table.
    """
    try:
        result = sync_students_service(db)
        return {
            "status": "success",
            "created": result["created"],
            "updated": result["updated"],
            "skipped": result["skipped"],
            "total": result["total"]
        }
    except Exception as exc:
        raise HTTPException(
            status_code=status.HTTP_500_INTERNAL_SERVER_ERROR,
            detail=f"Sync failed: {str(exc)}"
        )

Now let's explain each piece:

@router.post("/students") — this decorator does two things: (1) it registers the function as a handler for POST /students, and (2) it tells Swagger UI the HTTP method, URL, and description. The extra parameters summary and response_description are purely for documentation.

db: Session = Depends(get_db) — this is dependency injection. The Depends(get_db) tells FastAPI: "before calling this function, run get_db() and pass its result as db." The get_db() function opens a SQLAlchemy database session, yields it, and closes it when the request finishes. You never need to open or close the session yourself — FastAPI's dependency system handles the lifecycle entirely.

_user = Depends(require_any_role) — a second dependency that checks authentication. require_any_role reads the session cookie from the request, looks up the user in admin_users.json, and raises 401 Unauthorized if the cookie is missing or invalid. The underscore prefix in _user is a Python convention meaning "I declared this dependency for its side effect (auth check) but don't need to use the return value in this function."

result = sync_students_service(db) — calls the service function with the database session. The service does all the heavy lifting: calling Moodle API, parsing users, upserting each record, logging the sync event. It returns a summary dictionary.

raise HTTPException(...) — FastAPI's standard way to return an error response. When you raise this exception anywhere in the call chain, FastAPI catches it and returns a properly formatted JSON error response with the specified status code and detail message. Your endpoint function never needs to manually build error response dicts.

دعنا نشرح الأجزاء المهمة في هذه النقطة:

@router.post("/students") — هذا الديكوراتور يسجّل الدالة كمعالج لـ POST /students، ويخبر Swagger UI بطريقة HTTP والرابط والوصف.

db: Session = Depends(get_db) — هذا هو حقن التبعيات. Depends(get_db) يخبر FastAPI: "قبل استدعاء هذه الدالة، شغّل get_db() ومرّر نتيجتها كـ db." دالة get_db() تفتح جلسة SQLAlchemy وتغلقها تلقائياً عند انتهاء الطلب. لا تحتاج أبداً لفتح الجلسة أو إغلاقها يدوياً.

_user = Depends(require_any_role) — تبعية ثانية تتحقق من المصادقة. require_any_role تقرأ كوكي الجلسة وتتحقق من المستخدم. إذا كانت الكوكي مفقودة أو غير صالحة، تُطلق خطأ 401 Unauthorized.

raise HTTPException(...) — طريقة FastAPI القياسية لإعادة رد خطأ. عند إطلاق هذا الاستثناء، تمسكه FastAPI وتُعيد رداً JSON منسّقاً بشكل صحيح.

5 — What Is the Admin Panel?

٥ — ما هي لوحة التحكم؟

The Admin Panel is a full web interface accessible at http://localhost:8001/admin. It is not a Single-Page Application (SPA) built with React or Vue — it is a classic server-rendered multi-page application. This means:

  • When you navigate to /admin/dashboard, FastAPI receives the GET request, queries the database for the latest stats, renders an HTML file (using Jinja2), and sends the complete HTML page to your browser.
  • When you click "Sync Students", the browser sends a form POST to /admin/sync/students. FastAPI runs the sync, renders the result page (with success/error message), and sends the new HTML back — the whole page refreshes.

This approach has significant advantages for a management tool:

  • No frontend build step — no npm install, no webpack, no build artifacts. The HTML templates are source code.
  • Simpler authentication — session cookies work natively in browsers with server-rendered apps without complex token refresh logic.
  • Works without JavaScript — the core pages function even if JS is disabled (though some dynamic features like the log auto-refresh need it).
  • Easier to debug — "View Source" in the browser shows exactly what the server sent.

لوحة التحكم هي واجهة ويب كاملة في http://localhost:8001/admin. ليست تطبيق صفحة واحدة (SPA) مبنياً بـ React أو Vue — بل هي تطبيق متعدد الصفحات يُقدّم من الخادم (server-rendered).

عند التنقل إلى /admin/dashboard: تستقبل FastAPI طلب GET، تستعلم قاعدة البيانات عن أحدث الإحصائيات، تُقدّم ملف HTML (باستخدام Jinja2)، وترسل صفحة HTML كاملة للمتصفح. هذا النهج له مزايا: لا خطوة بناء frontend، مصادقة أبسط بكوكيات الجلسة، يعمل حتى بدون JavaScript، وأسهل في التشخيص.

٦ — صفحات لوحة التحكم — جولة مرئية

Here is what you see on each admin page after logging in:

/admin/login

A centered login form with two fields: "Username" and "Password", plus a "Login" button. The form POSTs to /admin/login. On success, FastAPI creates an encrypted session cookie and redirects to /admin/dashboard. On failure, the same page reloads with an error message: "Invalid username or password." There is no "Remember me" option — sessions expire when the browser is closed (configurable).

The first page after login. Contains a top row of summary cards showing: total students in the database, total enrolment records, total grade records, total payment records, and the timestamp of the last completed sync. Below the cards: a table of the 20 most recent sync events (type, status, number of records processed, timestamp). A sidebar on the left shows navigation links to all admin pages.

Four individual sync buttons and one "Full Sync" button:

  • Sync Students — pulls all user accounts from Moodle and upserts into the local students table.
  • Sync Grades — fetches all grade items for all enrolled students.
  • Sync Enrolments — refreshes course enrolment records.
  • Sync Payments — pulls payment/order records from Moodle.
  • Full Sync — runs all four syncs in sequence.

Each button is inside a form. When you click it, a POST request fires. The server runs the sync synchronously (the browser waits), then redirects back to this page with a flash message at the top showing the result: "Sync Students completed: 245 created, 12 updated, 3 skipped."

/admin/sync/history

A paginated table (20 rows per page) of all sync events ever logged, in reverse chronological order. Columns: event type, status (green checkmark for OK, red X for error), records processed, duration in seconds, timestamp, error message (if any). You can filter by event type using a dropdown at the top. Useful for diagnosing when syncs started failing.

A table showing the field mapping configuration. Each row represents one mapping: Zoho module (e.g., Contacts), Zoho field name (e.g., Email), local table (e.g., students), local column (e.g., email), and an optional transform function. A tabbed interface to browse the database contents directly. Tabs: Students, Enrolments, Grades, Payments. Each tab shows a searchable, sortable table. /admin/settings

A two-column form showing all editable .env keys on the left and input boxes on the right. Sensitive fields (like ZOHO_CLIENT_SECRET) show a masked value (***) with a reveal button. When you submit the form, FastAPI writes the new values back to the .env file. A yellow banner reminds you to restart the server for changes to take effect (since the Settings object is only read at startup).

Displays the last 200 lines from the server's log output in a dark-background code block. Lines are colour-coded by level: DEBUG (gray), INFO (white), WARNING (yellow), ERROR (red). A table listing all admin panel user accounts (from admin_users.json). Columns: username, role, date created. Actions: "Change Password" (shows a modal form), "Delete" (with confirmation prompt). A multi-step wizard for first-time configuration. 7 — How Authentication Works

٧ — كيف تعمل المصادقة؟

The admin panel uses session-based authentication — the classic, proven approach for web UIs that are rendered by the server. Here is the complete flow:

Step 1: Storing admin users

Admin users are stored in backend/admin_users.json — a JSON file listing usernames and their bcrypt-hashed passwords. bcrypt is a one-way hash function — there is no way to get the plain password back from the hash. To verify a login, you hash the input and compare it to the stored hash. 

[
  {
    "username": "admin",
    "password_hash": "$2b$12$abcdefghij...",
    "role": "admin"
  }
]

Step 2: The login flow

  1. User submits the login form — browser sends POST /admin/login with form fields username and password.
  2. auth.py reads admin_users.json, finds the matching username, and calls bcrypt.checkpw(password.encode(), stored_hash) to verify the password. If it doesn't match, returns the login page with an error.
  3. If the password matches, auth.py creates a signed, encrypted JWT (JSON Web Token) or a Fernet-encrypted session token containing the username and role, and sets it as an HttpOnly cookie named session.
  4. The browser stores this cookie and sends it automatically with every subsequent request to the admin panel.

Step 3: Protecting routes with require_any_role

Every admin panel page handler declares user = Depends(require_any_role). This FastAPI dependency function:

  1. Reads the session cookie from the request.
  2. Decrypts/verifies it.
  3. If invalid or missing, raises HTTPException(302) that redirects to /admin/login.
  4. If valid, returns the user dict so the route handler can access user["username"] if needed.

Adding a new admin user:

# From inside backend/ with the virtual environment active:
python -c "
import bcrypt, json
password = b'your_new_password_here'
hashed = bcrypt.hashpw(password, bcrypt.gensalt(rounds=12)).decode()
print(hashed)
"
# Copy that hash and manually add to admin_users.json:
# {"username": "newuser", "password_hash": "<pasted hash>", "role": "admin"}

# Or use the /admin/users page in the web UI — it handles this for you.

تستخدم لوحة التحكم المصادقة بالجلسات — النهج التقليدي والموثوق لتطبيقات الويب المُقدَّمة من الخادم.

الخطوة ١: يُخزَّن مستخدمو الإدارة في admin_users.json بكلمات مرور مُجزَّأة بـ bcrypt (دالة تجزئة أحادية الاتجاه — لا يمكن استعادة كلمة المرور من الهاش).

الخطوة ٢: عند تقديم نموذج الدخول، يقرأ auth.py الملف JSON ويتحقق من كلمة المرور بـ bcrypt.checkpw(). إذا تطابقا، يُنشئ توكن جلسة مشفّر ويضعه ككوكي HttpOnly يُرسله المتصفح تلقائياً مع كل طلب لاحق.

الخطوة ٣: كل معالج صفحة إدارة يُعلن user = Depends(require_any_role). هذه الدالة تقرأ كوكي الجلسة، تتحقق منها، وإذا كانت مفقودة/غير صالحة تُعيد توجيهاً إلى صفحة الدخول. إذا كانت صالحة، تُعيد بيانات المستخدم.

8 — Jinja2 Templates — How HTML Pages Are Built

٨ — قوالب Jinja2 — كيف تُبنى صفحات HTML

Jinja2 is a Python template engine — a system for generating HTML by mixing a static HTML skeleton with dynamic data. Think of a Jinja2 template as a document with "fill-in-the-blank" placeholders. You write the HTML structure once, and the server fills in the blanks with real data on every request.

Template syntax basics:

{# This is a comment — not rendered in the HTML #}

<!-- Print a variable -->
<p>Welcome, {{ user.username }}!</p>

<!-- Conditionals -->
{% if sync_result.status == "success" %}
  <div class="alert-success">Sync completed: {{ sync_result.created }} created.</div>
{% else %}
  <div class="alert-error">Sync failed: {{ sync_result.error }}</div>
{% endif %}

<!-- Loops -->
<table>
  <tbody>
    {% for student in students %}
    <tr>
      <td>{{ student.firstname }} {{ student.lastname }}</td>
      <td>{{ student.email }}</td>
    </tr>
    {% endfor %}
  </tbody>
</table>

How data flows from router.py to template:

In admin/router.py, every page handler ends with a TemplateResponse call: 

from fastapi.templating import Jinja2Templates
from fastapi import Request

templates = Jinja2Templates(directory="admin/templates")

@admin_router.get("/dashboard")
async def dashboard(
    request: Request,
    db: Session = Depends(get_db),
    user = Depends(require_any_role)
):
    student_count = db.query(Student).count()
    enrolment_count = db.query(Enrollment).count()
    recent_events = db.query(SyncEvent).order_by(
        SyncEvent.created_at.desc()
    ).limit(20).all()

    return templates.TemplateResponse(
        "dashboard.html",           # which template file to render
        {
            "request": request,     # always required by FastAPI Jinja2
            "user": user,           # current logged-in user
            "student_count": student_count,
            "enrolment_count": enrolment_count,
            "recent_events": recent_events,
        }
    )

The dictionary you pass as the second argument becomes the template context — every key in that dictionary is available as a variable in the Jinja2 template. The {{ student_count }} placeholder in dashboard.html will be replaced with the actual number when the template is rendered.

Template inheritance with base.html:

Every page in the admin panel shares the same layout: sidebar on the left, main content area on the right, a top bar with the page title and user info. Instead of copying this structure into every template file (which would be a maintenance nightmare), Jinja2 supports template inheritance. A parent template (base.html) defines the overall layout with a block placeholder: 

<!-- base.html -->
<!DOCTYPE html>
<html>
<head>
  <title>{% block title %}Admin{% endblock %} | Moodle-Zoho Admin</title>
  <link rel="stylesheet" href="/static/admin.css">
</head>
<body>
  <aside class="sidebar">
    <!-- navigation links -->
    <a href="/admin/dashboard">Dashboard</a>
    <a href="/admin/sync">Sync</a>
    ...
  </aside>
  <main>
    {% block content %}
    <!-- child templates fill this in -->
    {% endblock %}
  </main>
</body>
</html>

Each individual page template extends base.html and only defines its blocks:

<!-- dashboard.html -->
{% extends "base.html" %}

{% block title %}Dashboard{% endblock %}

{% block content %}
  <h1>Dashboard</h1>
  <div class="stat-card">
    <span class="stat-number">{{ student_count }}</span>
    <span class="stat-label">Students</span>
  </div>
  <!-- more content -->
{% endblock %}

When Jinja2 renders dashboard.html, it takes base.html's full HTML structure and inserts the {% block content %} section from dashboard.html. You get the complete page with the sidebar, header, and all the common chrome — by writing only the unique content of each page.

Jinja2 هو محرّك قوالب Python — نظام لتوليد HTML بخلط هيكل HTML ثابت مع بيانات ديناميكية. فكّر في قالب Jinja2 كوثيقة بـ "فراغات للتعبئة". تكتب هيكل HTML مرة واحدة، والخادم يملأ الفراغات ببيانات حقيقية عند كل طلب.

صيغة القوالب: {{ variable }} لطباعة متغير، {% if condition %}...{% endif %} للشروط، {% for item in list %}...{% endfor %} للحلقات.

كيف تتدفق البيانات من router.py إلى القالب: في router.py، كل معالج صفحة ينتهي بـ templates.TemplateResponse("dashboard.html", {...}). القاموس الذي تمرّره يصبح سياق القالب — كل مفتاح متاح كمتغير في Jinja2.

وراثة القوالب مع base.html: كل صفحة في لوحة التحكم تشترك في نفس التخطيط. بدلاً من نسخ هذا الهيكل في كل ملف قالب، يدعم Jinja2 وراثة القوالب. القالب الأب (base.html) يعرّف التخطيط الكلي مع لواصق {% block content %}. كل قالب صفحة فردي يمتدّ base.html ويعرّف لواصقه فقط. عند تقديم Jinja2 للقالب، يأخذ هيكل base.html الكامل ويُدرج قسم المحتوى الفريد للصفحة.
