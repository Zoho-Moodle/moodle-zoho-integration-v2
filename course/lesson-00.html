1 — What Is This Project?

١ — ما هو هذا المشروع؟

This project is an integration platform that connects two separate systems: Moodle LMS (the learning management system where students enrol in courses, receive grades, and make payments) and Zoho CRM (a customer-relationship platform where student records, deals, and contacts are managed by staff). Without this platform, staff would manually copy data between the two systems — a slow, error-prone process that breaks down completely at any real scale.

The backend is a Python 3.11 FastAPI web server that sits between Moodle and Zoho. It exposes REST API endpoints that Moodle calls (via a PHP plugin installed on the Moodle server), and it also calls Zoho's API to read and write CRM records. All data is cached locally in a SQLite database called moodle_zoho_local.db which lives inside the backend/ folder. The server listens on port 8001.

This course teaches you every layer of this stack — from setting up your local machine all the way to understanding how a live sync event flows from a Moodle grade change into a Zoho CRM field update. No prior knowledge of FastAPI, SQLAlchemy, or Zoho APIs is assumed.

هذا المشروع هو منصة تكامل تربط بين نظامين منفصلين: Moodle LMS (نظام إدارة التعلّم الذي يسجّل فيه الطلاب في الدورات ويحصلون على الدرجات ويدفعون الرسوم) وZoho CRM (منصة إدارة علاقات العملاء التي تحتفظ بسجلات الطلاب والصفقات وبيانات التواصل). بدون هذه المنصة، سيضطر الموظف إلى نسخ البيانات يدوياً بين النظامين — عملية بطيئة وعرضة للأخطاء ومستحيلة على نطاق واسع.

الخادم الخلفي (backend) هو خادم Python 3.11 FastAPI يجلس بين Moodle وZoho. يوفّر نقاط نهاية REST API يمكن لـ Moodle استدعاؤها (عبر إضافة PHP مثبّتة على خادم Moodle)، كما يستدعي Zoho API لقراءة وكتابة سجلات CRM. تُخزَّن البيانات محلياً في قاعدة بيانات SQLite تسمى moodle_zoho_local.db داخل مجلد backend/. يعمل الخادم على المنفذ 8001.

2 — What Is Python?

٢ — ما هو Python؟

Python is a general-purpose programming language designed to be readable and fast to write. It looks closer to plain English than most languages, which makes it popular for beginners, data scientists, and web developers alike. When you write print("Hello") in Python, it actually prints "Hello" — no boilerplate, no semicolons required.

This project requires Python 3.11 specifically. The version matters because:

  • Python 3.11 introduced improved exception messages that show exactly which variable caused an error.
  • Some libraries in requirements.txt declared minimum version compatibility of 3.11.
  • The SQLAlchemy 2.x type annotations used throughout the codebase require 3.10+.

To check which version is installed, open a terminal and type:

python --version
# Expected:  Python 3.11.x

# On some systems you may need:
python3 --version

If you see Python 3.8 or older — or if the command is not found — install Python 3.11:

  • Windows: Download the installer from https://python.org/downloads. During installation, tick "Add Python to PATH" — without this, your terminal won't find the python command.
  • macOS: brew install python@3.11 (requires Homebrew).
  • Ubuntu/Debian: sudo apt install python3.11 python3.11-venv.
Windows PATH gotcha: If the installer finishes successfully but python --version still fails, close your terminal, open a new one, and try again. Windows reads the PATH environment variable only at terminal startup.

Python هو لغة برمجة عامة الأغراض صُمِّمت لتكون مقروءة وسريعة الكتابة. تبدو أقرب إلى الإنجليزية السادة مقارنة بمعظم اللغات، مما يجعلها شائعة لدى المبتدئين وعلماء البيانات ومطوري الويب على حدٍّ سواء.

يتطلب هذا المشروع Python 3.11 تحديداً لأن بعض الميزات والمكتبات تعتمد على هذا الإصدار. للتحقق من الإصدار المثبّت: python --version — يجب أن يظهر Python 3.11.x. إذا ظهر إصدار أقدم أو ظهرت رسالة خطأ، ثبّت Python 3.11 من python.org (ويندوز) وتأكد من تحديد خيار "Add Python to PATH" أثناء التثبيت.

3 — What Is a Virtual Environment?

٣ — ما هي البيئة الافتراضية (Virtual Environment)؟

Imagine you have two projects on your computer. Project A needs version 1.0 of a library, but Project B needs version 2.0 of the same library. If you install both versions globally on your system Python, they will conflict — only one can be "installed" at a time, and whichever project runs last wins. The other project breaks.

A virtual environment solves this elegantly. Think of it like giving every project its own separate backpack. Each backpack holds its own copy of Python and its own collection of installed libraries. Project A's backpack has version 1.0; Project B's backpack has version 2.0. They never interfere with each other because the computer always checks the current project's backpack first.

In Python, you create a virtual environment with the built-in venv module:

# Run this from the project root directory
python -m venv .venv-2

This command creates a new folder called .venv-2 at the location you run it from. Inside, Python copies a minimal Python interpreter and pip. On Windows, the folder structure is: 

.venv-2/
├── Scripts/
│   ├── python.exe      ← the private Python interpreter
│   ├── pip.exe         ← the private pip
│   └── Activate.ps1    ← the activation script for PowerShell
├── Lib/
│   └── site-packages/  ← installed packages go here
└── pyvenv.cfg          ← config file pointing to the base Python

Why the name .venv-2? It's a deliberate convention in this project to avoid confusion with any older .venv folder that may exist from a previous setup attempt. The leading dot makes it a hidden folder on Linux/macOS (cleaner directory listings). The name is arbitrary — what matters is that it is listed in .gitignore so that this machine-local binary folder is never committed to the git repository.

تخيّل أنك تملك مشروعين. المشروع A يحتاج الإصدار 1.0 من مكتبة معينة، والمشروع B يحتاج الإصدار 2.0 من نفس المكتبة. لو ثبّتّهما في Python العالمي على جهازك سيتعارضان.

البيئة الافتراضية تحلّ هذا بمنح كل مشروع حقيبة ظهر خاصة به. كل حقيبة تحتوي نسختها الخاصة من Python ومجموعتها الخاصة من المكتبات. حقيبة المشروع A فيها الإصدار 1.0؛ حقيبة المشروع B فيها الإصدار 2.0. لا تتداخلان أبداً.

في Python، تنشئ بيئة افتراضية بوحدة venv المدمجة: python -m venv .venv-2 هذا ينشئ مجلد .venv-2 يحتوي نسخة Python خاصة وpip خاص وحزم مثبّتة معزولة. لماذا الاسم .venv-2؟ اتفاقية في هذا المشروع تُميّزه عن أي .venv قديمة. الاسم اختياري لكنه مدرج في .gitignore حتى لا يُرفع إلى مستودع الكود.

4 — Activating the Virtual Environment

٤ — تفعيل البيئة الافتراضية

Creating the virtual environment folder is just the first step. You must also activate it in your terminal session. Activation sets several environment variables (primarily PATH) so that when you type python or pip, your terminal uses the private copies inside .venv-2/ rather than the system-wide Python.

Activation commands by platform:

Windows — PowerShell (recommended):

# Using a relative path (from the project root):
& .\.venv-2\Scripts\Activate.ps1

# Or using the full absolute path:
& "c:\Users\MohyeddineFarhat\Documents\GitHub\moodle-zoho-integration-v3\.venv-2\Scripts\Activate.ps1"

Windows — Command Prompt (cmd.exe):

.venv-2\Scripts\activate.bat

Linux / macOS — bash or zsh:

source .venv-2/bin/activate

After successful activation, your terminal prompt changes to include the environment name in parentheses at the start: 

(.venv-2) PS C:\Users\MohyeddineFarhat\Documents\GitHub\moodle-zoho-integration-v3>

That (.venv-2) prefix is your constant reminder and confirmation that the virtual environment is active. Every python and pip command now uses the project-local versions. To deactivate the environment and return to the system Python, simply type deactivate.

Windows ExecutionPolicy Error: If PowerShell refuses to run Activate.ps1 with a message like "...cannot be loaded because running scripts is disabled on this system", you need to relax the execution policy. Run this once in PowerShell (no administrator required for CurrentUser scope): 
Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser
RemoteSigned means: local scripts can run freely; downloaded scripts must be digitally signed by a trusted publisher. This is a safe and common setting for developer machines.

إنشاء مجلد البيئة الافتراضية ليس كافياً — يجب تفعيلها في جلسة التيرمينال الحالية. التفعيل يضبط متغيرات البيئة (خاصةً PATH) لكي يستخدم التيرمينال النسخ الخاصة من Python وpip الموجودة داخل .venv-2/ بدلاً من Python المثبّت على مستوى النظام.

على ويندوز PowerShell: & .\.venv-2\Scripts\Activate.ps1 على لينوكس/macOS: source .venv-2/bin/activate بعد التفعيل يتغيّر سطر التيرمينال ليظهر (.venv-2) في البداية — تأكيد مرئي أن البيئة نشطة. لإلغاء التفعيل والعودة لـ Python النظام: اكتب deactivate.

خطأ ExecutionPolicy على ويندوز: إذا رفض PowerShell تشغيل Activate.ps1، شغّل هذا الأمر مرة واحدة (لا يحتاج صلاحيات مدير): Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser

5 — Installing Dependencies

٥ — تثبيت التبعيات (Dependencies)

pip (Pip Installs Packages) is Python's official package manager. It downloads libraries from PyPI (the Python Package Index — think of it as an app store for Python libraries with over 500,000 packages) and installs them into your active virtual environment.

A requirements.txt file is a plain text document listing every package your project depends on, one per line, optionally with a pinned version number (e.g., fastapi==0.111.0). Pinning versions ensures everyone on the team — and every deployment environment — installs exactly the same version, preventing "works on my machine" bugs.

Install everything with one command (run from the project root, with the venv active):

pip install -r backend/requirements.txt

The -r flag means "read the requirements from this file". pip reads every line and installs each package and its own sub-dependencies in turn. You will see a stream of download and installation messages. When it finishes, type pip list to see all installed packages.

Key packages in this project and what they do:

sqlalchemy ORM ORM = Object-Relational Mapper. Instead of writing raw SQL like SELECT * FROM students, you write Python like db.query(Student).all(). SQLAlchemy translates Python objects into SQL statements and back.

6 — The .env File — Secrets and Configuration

٦ — ملف الإعدادات السرية (.env)

A .env file is a plain text file that stores environment variables — key=value pairs that configure how your application behaves. Variables fall into two categories:

  • Configuration — values that differ between environments, like LOG_LEVEL=DEBUG locally vs LOG_LEVEL=WARNING in production.
  • Secrets — sensitive credentials like API keys, tokens, and cryptographic keys that must never be shared publicly.

Why are secrets not committed to Git? Once something is committed to a git repository and pushed, it is in the git history permanently — even if you delete the file in a later commit, an attacker can check out old commits and find it. If the repository is public or is ever made public, your Zoho credentials, Moodle token, and encryption key are all compromised instantly.

The solution: a .env file is listed in .gitignore so git never sees it. A safe template called .env.example is committed instead — it lists all variable names with placeholder or empty values, serving as documentation of what you need to fill in.

Create your .env from the example:

# Windows PowerShell:
Copy-Item backend\.env.example backend\.env

# Windows Command Prompt:
copy backend\.env.example backend\.env

# Linux / macOS:
cp backend/.env.example backend/.env

# Then open and fill in your real values:
notepad backend\.env   # Windows
nano backend/.env      # Linux/macOS

Every variable in this project explained:

VariableExample ValueExplanation
DATABASE_URL sqlite:///./moodle_zoho_local.db SQLAlchemy database connection string. The format is dialect:///path. The sqlite:///./ prefix means "SQLite file in the current working directory". The ./ is relative — this is the entire reason you must cd backend before starting the server. If you start from the project root, SQLAlchemy looks for the .db file in the wrong folder.
APP_NAME Moodle-Zoho Integration v3 Human-readable application name used in the Swagger UI title bar and the Admin Panel header.
ENV development Runtime environment flag. Set to development locally and production on the live server. The app uses this to toggle features like detailed error tracebacks (shown in development, hidden in production).
LOG_LEVEL DEBUG Verbosity of server logs. DEBUG prints everything including routine HTTP requests. INFO prints normal flow. WARNING and ERROR only print problems. Use DEBUG locally and INFO or WARNING in production.
ENCRYPTION_KEY (32-byte base64 key) A Fernet symmetric encryption key used to encrypt sensitive values before storing them in the SQLite database. Generate a fresh key with: python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())". Keep this secret — losing it means you cannot decrypt stored values.
MOODLE_BASE_URL https://your-moodle.com The root URL of your Moodle installation (no trailing slash). The backend appends /webservice/rest/server.php to build every API call to Moodle.
MOODLE_TOKEN (40-char hex string) A Moodle Web Services token granting API access to read students, courses, grades, and enrolments. Created in Moodle under Site Administration → Web Services → Manage tokens. The token is associated with a specific Moodle user account that must have the required web service permissions.
ZOHO_CLIENT_ID (from Zoho Developer Console) OAuth2 Client ID for the server-based Zoho application. Created at https://accounts.zoho.com/developerconsole. This identifies your application to Zoho's OAuth2 system.
ZOHO_CLIENT_SECRET (secret) The secret paired with the Client ID. Never share this or commit it. Together with the Client ID, this proves to Zoho that your server is the legitimate application — not an impersonator.
ZOHO_REFRESH_TOKEN (from OAuth2 grant) A long-lived OAuth2 token used to obtain fresh short-lived access tokens without requiring a user to log in again. Obtained once via the OAuth2 authorization code flow, then stored here permanently. The backend uses it automatically behind the scenes.

ملف .env هو ملف نصي يحتوي على أزواج مفتاح=قيمة لتهيئة تطبيقك. ينقسم المتغيرات إلى قسمين: إعدادات (قيم تختلف بين البيئات) وأسرار (بيانات حساسة كمفاتيح API والتوكنات التي يجب ألا تُشارك علنياً).

لماذا لا نرفع الأسرار إلى Git؟ لأن أي شيء يُرفع إلى مستودع git يبقى في تاريخه للأبد — حتى لو حذفت الملف لاحقاً، يمكن لأي شخص استعادة النسخ القديمة ورؤية الأسرار. الحل: إدراج .env في .gitignore ورفع ملف آمن يسمى .env.example يحتوي أسماء المتغيرات بقيم وهمية فقط.

7 — Starting the Server

٧ — تشغيل الخادم

Two rules before every server start: (1) the virtual environment must be active, and (2) your working directory must be backend/.

Why must you be inside backend/? The DATABASE_URL in your .env is sqlite:///./moodle_zoho_local.db. The ./ means "current directory when the server starts". The .db file lives at backend/moodle_zoho_local.db. If you start the server from the project root, Python looks for (and attempts to create) the database in the project root instead — in the wrong folder, with wrong relative imports. 

# Full startup sequence from the project root:

# Step 1: activate venv (skip if already showing (.venv-2) prompt)
& .\.venv-2\Scripts\Activate.ps1

# Step 2: move into backend
cd backend

# Step 3: launch the server
python start_server.py

What start_server.py does:

It calls uvicorn.run("app.main:app", host="0.0.0.0", port=8001, reload=False) which imports app/main.py, creates the FastAPI application object, binds it to all network interfaces on port 8001, and starts the event loop.

Expected startup output (healthy server):

INFO:     Started server process [12345]
INFO:     Waiting for application startup.
INFO:     Creating database tables if not exist...
INFO:     Database tables ready.
INFO:     Scheduler started. Next sync in 15 minutes.
INFO:     Application startup complete.
INFO:     Uvicorn running on http://0.0.0.0:8001 (Press CTRL+C to quit)

URLs available once running:

  • http://localhost:8001 — API root. Returns a JSON status object: {"name": "Moodle-Zoho Integration v3", "status": "running", "version": "3.0.0"}
  • http://localhost:8001/docs — Swagger UI: fully interactive API documentation. Every endpoint is listed with its parameters. You can send test requests directly from your browser — no Postman or curl required.
  • http://localhost:8001/redoc — ReDoc: alternative API docs in a cleaner read-only format, useful for sharing documentation with non-technical stakeholders.
  • http://localhost:8001/admin — Admin Panel: the full web-based management interface. Requires login (see lesson-02 for the admin panel walkthrough).
  • http://localhost:8001/health — Health check: returns {"status": "ok", "database": "connected"} and is used by deployment monitoring tools (systemd, uptime monitors) to verify the service is alive.
Keyboard shortcut: Press Ctrl+C in the terminal to stop the server gracefully. It will finish any in-progress request before shutting down.

قبل تشغيل الخادم، قاعدتان: (١) البيئة الافتراضية يجب أن تكون نشطة، (٢) مجلد العمل يجب أن يكون backend/.

لماذا يجب الانتقال إلى backend/؟ لأن DATABASE_URL في .env يحتوي sqlite:///./moodle_zoho_local.db والـ ./ تعني "المجلد الحالي عند تشغيل الخادم". ملف قاعدة البيانات موجود في backend/moodle_zoho_local.db. لو شغّلت الخادم من المجلد الجذر، سيبحث SQLAlchemy عن قاعدة البيانات في المكان الخطأ. 

cd backend
python start_server.py

عند بدء التشغيل الناجح ستظهر رسالة Uvicorn running on http://0.0.0.0:8001. يمكنك بعدها الوصول إلى: /docs (توثيق تفاعلي)، /admin (لوحة التحكم)، /health (التحقق من السلامة). لإيقاف الخادم: Ctrl+C.

8 — The SQLite Database

٨ — قاعدة بيانات SQLite

SQLite is a serverless, embedded relational database engine. Unlike MySQL or PostgreSQL — which run as separate server processes you connect to over a socket or network — SQLite stores the entire database in a single ordinary file on disk. The Python standard library includes a sqlite3 module, so no external software needs to be installed at all.

In this project the database file is backend/moodle_zoho_local.db. On first startup, if this file does not exist, SQLAlchemy runs: 

Base.metadata.create_all(bind=engine)

This inspects every Python class decorated with @mapper_registry.mapped or inheriting from Base (defined in backend/app/infra/db/base.py) and creates the corresponding SQL tables if they don't already exist. You never have to write a CREATE TABLE statement manually during development.

Why SQLite (and not PostgreSQL or MySQL)?

  • Zero installation: No database server to install, configure, or keep running. Perfect for local development and for a single-server deployment.
  • Zero configuration: No users, passwords, ports, or connection pools to manage for the database itself.
  • Portability: The entire database is a single file — copy it to another machine and everything works. Useful for backups and migrations.
  • Sufficient for this workload: Syncing a few thousand students and grades per day is well within SQLite's capabilities. It handles concurrent reads well and is used in production by many systems handling millions of rows.
No Docker, No PostgreSQL: This project deliberately does not use Docker or PostgreSQL. The DATABASE_URL is always a SQLite file path. If you see any documentation referring to postgres:// — it does not apply to this project.

Inspecting the database from Python:

import sqlite3

# Run this script from inside backend/ so the relative path works
conn = sqlite3.connect("moodle_zoho_local.db")
cur = conn.cursor()

# See all tables:
cur.execute("SELECT name FROM sqlite_master WHERE type='table' ORDER BY name;")
print("Tables:", [row[0] for row in cur.fetchall()])

# Count students:
cur.execute("SELECT COUNT(*) FROM students;")
print("Total students:", cur.fetchone()[0])

# Preview first 3 students:
cur.execute("SELECT id, moodle_id, firstname, lastname, email FROM students LIMIT 3;")
for row in cur.fetchall():
    print(row)

conn.close()

For a more comfortable experience, use a free GUI tool called DB Browser for SQLite (available at https://sqlitebrowser.org). Open backend/moodle_zoho_local.db in it and you get a full table browser, SQL editor, and data explorer — no command-line needed.

SQLite هو محرك قاعدة بيانات علائقية مدمج ولا يحتاج خادماً منفصلاً. بخلاف MySQL أو PostgreSQL (اللذَين يعملان كعملية خادم منفصلة)، يخزّن SQLite قاعدة البيانات كاملةً في ملف واحد على القرص. مكتبة Python القياسية تتضمّن sqlite3، لذا لا حاجة لتثبيت أي برنامج خارجي.

في هذا المشروع يوجد الملف هنا: backend/moodle_zoho_local.db. عند أول تشغيل، إذا لم يكن الملف موجوداً، يشغّل SQLAlchemy Base.metadata.create_all(bind=engine) الذي يفحص كل نماذج Python وينشئ الجداول المقابلة تلقائياً. لا تحتاج أبداً لكتابة CREATE TABLE يدوياً.

لماذا SQLite (وليس PostgreSQL أو MySQL)؟ لأنه لا يحتاج تثبيتاً أو إعداداً، قابل للنقل كملف واحد، وكافٍ لعبء العمل (مزامنة آلاف الطلاب والدرجات يومياً). هذا المشروع لا يستخدم Docker ولا PostgreSQL — رابط قاعدة البيانات دائماً مسار ملف SQLite.

لفحص قاعدة البيانات استخدم وحدة sqlite3 المدمجة في Python (كما في المثال أعلاه)، أو استخدم أداة رسومية مجانية مثل DB Browser for SQLite من sqlitebrowser.org.

Quick Reference — Full Setup Sequence

المرجع السريع — التسلسل الكامل للإعداد

# ── 1. Verify Python version ──
python --version          # Must show Python 3.11.x

# ── 2. Create the virtual environment ──
python -m venv .venv-2

# ── 3. Activate it (Windows PowerShell) ──
& .\.venv-2\Scripts\Activate.ps1
# Prompt should now start with (.venv-2)

# ── 4. Install all dependencies ──
pip install -r backend/requirements.txt

# ── 5. Create .env from the example ──
Copy-Item backend\.env.example backend\.env
# Open backend\.env and fill in ALL variables

# ── 6. Start the server ──
cd backend
python start_server.py

# ── 7. Verify it works ──
# Open: http://localhost:8001/health   → {"status":"ok"}
# Open: http://localhost:8001/docs     → Swagger UI
# Open: http://localhost:8001/admin    → Admin Panel login

هذا هو التسلسل الكامل من الصفر. تذكّر دائماً: (١) تفعيل البيئة الافتراضية قبل أي أمر pip أو python. (٢) الانتقال إلى مجلد backend/ قبل تشغيل start_server.py. (٣) ملء جميع متغيرات .env قبل التشغيل.
