Lesson 09

Deployment to a Linux Server

النشر على سيرفر Linux

This lesson walks through the complete process of deploying the FastAPI backend from your development machine to a Linux production server — transferring files, setting up Python, configuring systemd, and setting up Nginx as a reverse proxy.

يستعرض هذا الدرس العملية الكاملة لنشر الباك-إند FastAPI من جهازك إلى سيرفر Linux للإنتاج — نقل الملفات، إعداد Python، ضبط systemd، وإعداد Nginx كـ reverse proxy.

1. What Does "Deployment" Mean?

١. ماذا يعني "النشر"؟

During development you run the server on your own laptop (localhost:8001). Only you can access it. Deployment means moving the code to a server that is connected to the internet 24/7 so Zoho and Moodle can reach it, and so users can access the Admin Panel from anywhere.

أثناء التطوير تُشغّل السيرفر على لابتوبك (localhost:8001). أنت وحدك تصل إليه. النشر يعني نقل الكود إلى سيرفر متصل بالإنترنت 24/7 حتى يستطيع Zoho و Moodle الوصول إليه، وليتمكن المستخدمون من الوصول للوحة التحكم من أي مكان.

Development:                    Production:
  Your laptop                     Linux VPS (Ubuntu 22.04)
  localhost:8001  ──────►         your-server.com:80/443
  only YOU can access             The internet can access
  you START/STOP it               Runs 24/7 via systemd service
  changes break things            Stable, tested code only

2. Prerequisites — What You Need

٢. المتطلبات المسبقة

  • A Linux VPS (Virtual Private Server) — Ubuntu 22.04 recommended. Providers: DigitalOcean, Hetzner, Linode (budget: $6-12/month)
  • A domain name pointing to your server's IP (e.g., sync.mycollege.com)
  • SSH access to the server (username + password or SSH key)
  • Git installed on your local machine

تحتاج: سيرفر VPS بنظام Ubuntu 22.04 (من DigitalOcean أو Hetzner أو Linode، بحوالي 6-12 دولار شهرياً)، اسم نطاق يشير لـ IP السيرفر، صلاحية SSH للوصول، وGit على جهازك.

3. Step 1 — Transfer Files to the Server

٣. الخطوة الأولى — نقل الملفات إلى السيرفر

Use rsync (or scp) to copy the project to the server. We exclude the virtual environment (too large), the .env file (create it fresh on the server), and the SQLite database (a new one will be created).

استخدم rsync لنسخ المشروع إلى السيرفر. نستثني البيئة الافتراضية (حجمها كبير)، ملف .env (سننشئه من جديد على السيرفر)، وقاعدة البيانات (ستُنشأ تلقائياً).

# Run this on your LOCAL machine (Windows: use Git Bash or WSL)
rsync -avz \
  --exclude=".venv-2" \
  --exclude=".env" \
  --exclude="*.db" \
  --exclude="__pycache__" \
  --exclude=".git" \
  ./backend/ \
  ubuntu@your-server-ip:/home/ubuntu/moodle-zoho-backend/

# Verify the transfer
ssh ubuntu@your-server-ip "ls /home/ubuntu/moodle-zoho-backend/"

4. Step 2 — Server Setup: Python & Virtual Environment

٤. الخطوة الثانية — إعداد Python والبيئة الافتراضية على السيرفر

# SSH into your server
ssh ubuntu@your-server-ip

# Update system packages
sudo apt update && sudo apt upgrade -y

# Install Python 3.11 (Ubuntu 22.04 has Python 3.10 by default)
sudo apt install -y python3.11 python3.11-venv python3.11-dev

# Navigate to the project
cd /home/ubuntu/moodle-zoho-backend

# Create the virtual environment
python3.11 -m venv .venv-2

# Activate it
source .venv-2/bin/activate

# Install all dependencies
pip install --upgrade pip
pip install -r requirements.txt

# Verify FastAPI is installed
python -c "import fastapi; print('FastAPI OK:', fastapi.__version__)"

قم بتسجيل الدخول إلى السيرفر عبر SSH، ثم نفّذ الأوامر أعلاه بالترتيب. سيستغرق pip install -r requirements.txt بضع دقائق أول مرة.

5. Step 3 — Create the .env File on the Server

٥. الخطوة الثالثة — إنشاء ملف .env على السيرفر

Never copy your development .env to the server — use a clean production version. Create it directly on the server with production values:

لا تنسخ ملف .env من جهازك إلى السيرفر — أنشئ نسخة إنتاج نظيفة مباشرةً على السيرفر:

nano /home/ubuntu/moodle-zoho-backend/.env

Add all required variables (same keys as your local .env, but production values):

SECRET_KEY=generate-a-long-random-string-here
DATABASE_URL=sqlite:///./moodle_zoho_local.db

ZOHO_CLIENT_ID=your-production-zoho-client-id
ZOHO_CLIENT_SECRET=your-production-zoho-client-secret
ZOHO_REFRESH_TOKEN=your-production-zoho-refresh-token
ZOHO_API_DOMAIN=https://www.zohoapis.eu
ZOHO_ACCOUNTS_DOMAIN=https://accounts.zoho.eu

MOODLE_URL=https://your-moodle.example.com
MOODLE_TOKEN=your-production-moodle-token

WEBHOOK_SECRET=a-different-secret-from-dev
ADMIN_USERNAME=admin
ADMIN_PASSWORD=use-a-strong-password-here

Save with Ctrl+O, exit with Ctrl+X. Then restrict file permissions so only root can read it:

chmod 600 /home/ubuntu/moodle-zoho-backend/.env

احفظ الملف وقيّد صلاحياته. لا تستخدم نفس كلمات المرور من بيئة التطوير. احفظ كلمة المرور في مدير كلمات مرور آمن.

6. Step 4 — systemd Service (Run 24/7 & Auto-restart)

٦. الخطوة الرابعة — خدمة systemd (تشغيل 24/7 وإعادة تشغيل تلقائية)

systemd is Linux's service manager. By creating a service unit file for our backend, it will: start automatically when the server boots, restart if it crashes, and keep running 24/7.

systemd هو مدير خدمات Linux. بإنشاء ملف وحدة خدمة للباك-إند، سيبدأ تلقائياً عند بدء تشغيل السيرفر، يُعيد التشغيل عند التعطل، ويعمل 24/7.

sudo nano /etc/systemd/system/moodle-zoho.service
[Unit]
Description=Moodle-Zoho Integration Backend
After=network.target

[Service]
User=ubuntu
Group=ubuntu
WorkingDirectory=/home/ubuntu/moodle-zoho-backend
Environment="PATH=/home/ubuntu/moodle-zoho-backend/.venv-2/bin"
ExecStart=/home/ubuntu/moodle-zoho-backend/.venv-2/bin/python start_server.py
Restart=on-failure          ; restart if the process crashes
RestartSec=10               ; wait 10 seconds before restarting
StandardOutput=journal      ; logs go to journald
StandardError=journal

[Install]
WantedBy=multi-user.target
# Enable and start the service
sudo systemctl daemon-reload             # reload service files
sudo systemctl enable moodle-zoho        # auto-start on boot
sudo systemctl start moodle-zoho         # start now
sudo systemctl status moodle-zoho        # check it is running

# View live logs
sudo journalctl -u moodle-zoho -f        # -f means follow (live tail)

بعد تشغيله، systemctl status يجب أن يظهر active (running) باللون الأخضر. إذا ظهر فشل، journalctl -u moodle-zoho -n 50 يعرض آخر 50 سطر من السجل لتشخيص المشكلة.

7. Step 5 — Nginx Reverse Proxy

٧. الخطوة الخامسة — Nginx كـ Reverse Proxy

Our FastAPI server runs on port 8001. Nginx runs on port 80 (HTTP) and 443 (HTTPS) and forwards traffic to our backend. This also lets us add SSL (HTTPS) using Let's Encrypt (free).

سيرفر FastAPI يعمل على المنفذ 8001. Nginx يعمل على المنفذ 80 (HTTP) و443 (HTTPS) ويُحوّل الطلبات إلى الباك-إند. هذا يسمح أيضاً بإضافة SSL (HTTPS) مجاناً عبر Let's Encrypt.

sudo apt install -y nginx
sudo nano /etc/nginx/sites-available/moodle-zoho
server {
    listen 80;
    server_name sync.your-domain.com;   # your domain name

    # Forward all requests to FastAPI
    location / {
        proxy_pass         http://127.0.0.1:8001;
        proxy_set_header   Host $host;
        proxy_set_header   X-Real-IP $remote_addr;
        proxy_set_header   X-Forwarded-For $proxy_add_x_forwarded_for;

        # Increase timeout for long sync operations
        proxy_read_timeout 120s;
    }
}
# Enable the site
sudo ln -s /etc/nginx/sites-available/moodle-zoho /etc/nginx/sites-enabled/
sudo nginx -t           # test config for syntax errors
sudo systemctl reload nginx

# Add free HTTPS with Let's Encrypt
sudo apt install -y certbot python3-certbot-nginx
sudo certbot --nginx -d sync.your-domain.com
# Follow prompts — certbot auto-updates the Nginx config for HTTPS!

بعد إعداد Nginx و SSL، يمكنك الوصول للوحة التحكم على https://sync.your-domain.com/admin من أي مكان في العالم.

8. Using ngrok for Webhook Testing

٨. استخدام ngrok لاختبار الـ Webhooks

During development, your localhost:8001 is not accessible by Zoho or Moodle. ngrok creates a temporary public URL that tunnels to your local server — perfect for testing webhooks before deploying.

أثناء التطوير، localhost:8001 غير متاح لـ Zoho أو Moodle. ngrok ينشئ رابطاً عاماً مؤقتاً يُحوّل إلى سيرفرك المحلي — مثالي لاختبار الـ webhooks قبل النشر.

# Download ngrok from https://ngrok.com/download (free account)
# Start your FastAPI server first, then in a second terminal:
ngrok http 8001

# You will see something like:
# Forwarding  https://abc123.ngrok.io -> http://localhost:8001

# Copy that https URL and set it as:
# - Backend URL in Moodle plugin settings
# - Webhook notification URL in Zoho Workflow Rules
# Note: URL changes every time you restart ngrok (free plan)
ngrok is for development/testing only. Do not use it in production. In production, use your actual domain with Nginx + Let's Encrypt SSL as described above.
تحذير: ngrok للتطوير والاختبار فقط. لا تستخدمه في الإنتاج. في الإنتاج، استخدم نطاقك الفعلي مع Nginx + Let's Encrypt كما شُرح أعلاه.

9. Production Checklist

٩. قائمة تحقق الإنتاج

  • ✅ ملف .env يحتوي قيم إنتاج حقيقية (ليس نسخة التطوير)
  • ✅ .env has chmod 600 permissions
  • ✅ كلمة المرور في ADMIN_PASSWORD قوية وفريدة
  • ✅ SQLite database file has regular backups (daily cron job to copy to safe location)
  • ✅ HTTPS مُفعَّل عبر Let's Encrypt وشهادة مجددة تلقائياً
  • ✅ systemd service is enabled (starts on boot)
  • ✅ تم اختبار واجهة /admin من متصفح خارجي على النطاق الحقيقي
  • ✅ Zoho webhook URL updated to production URL (not ngrok or localhost)
  • ✅ Moodle plugin webhook URL محدَّث برابط السيرفر الإنتاجي
  • ✅ A full sync triggered manually after deployment to verify everything works
# Simple daily backup cron job — adds to crontab
crontab -e
# Add this line:
0 3 * * * cp /home/ubuntu/moodle-zoho-backend/moodle_zoho_local.db \
    /home/ubuntu/backups/moodle_zoho_$(date +\%Y\%m\%d).db

الاحتياطي اليومي مهم جداً. قاعدة البيانات SQLite ملف واحد — نسخه إلى مكان آمن (أو تحميله إلى مخزن سحابي) كل يوم يحمي من فقدان البيانات.
