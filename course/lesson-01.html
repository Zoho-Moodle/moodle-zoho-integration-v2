Project Structure Deep Dive

Understand every folder and file in the repository — what it does, why it exists, and how it connects to the rest of the system.

هيكل المشروع بالتفصيل — كل مجلد وكل ملف

1 — Why Project Structure Matters

١ — لماذا يهمّنا هيكل المشروع؟

Imagine two offices. The first office has every document piled randomly on a single desk — contracts mixed with invoices mixed with HR forms. Finding anything takes ten minutes, and new employees have no idea where to start. The second office has labelled filing cabinets: one for contracts, one for invoices, one for HR. Any person — including someone who has never been there before — can find what they need in thirty seconds.

A software project without deliberate structure is the first office. When a bug appears in production at 2am, you cannot afford to spend twenty minutes figuring out which file handles which behaviour. Good project structure means every file has exactly one well-defined job, files that work together are grouped into the same folder, and any developer can navigate the codebase on their first day.

This project follows the layered architecture pattern, also called n-tier architecture. Each layer has a specific responsibility and only communicates with the layers immediately adjacent to it:

  • API layer (app/api/) — receives HTTP requests, validates input, returns HTTP responses.
  • Service layer (app/services/) — orchestrates business logic, calls infrastructure.
  • Infrastructure layer (app/infra/) — communicates with external systems (database, Moodle, Zoho).
  • Domain layer (app/domain/) — defines the data shapes shared across all layers.

تخيّل مكتبَين. المكتب الأول: كل وثيقة مرمية عشوائياً على طاولة واحدة — عقود مختلطة بفواتير مختلطة بملفات مستخدمين. إيجاد أي شيء يستغرق عشر دقائق. المكتب الثاني: خزانات ملفات مسمّاة: واحدة للعقود، واحدة للفواتير، واحدة للموارد البشرية. أي شخص — حتى المبتدئ — يجد ما يحتاجه خلال ثلاثين ثانية.

مشروع برمجي بدون هيكل متعمّد هو المكتب الأول. هيكل المشروع الجيد يعني أن كل ملف له وظيفة واحدة محددة بوضوح، والملفات التي تعمل معاً مجمّعة في نفس المجلد، وأي مطوّر يمكنه التنقّل في الكود من أول يوم.

يتبع هذا المشروع نمط الهندسة طبقية: طبقة API تستقبل الطلبات وتُعيد الردود، طبقة خدمة تنسّق منطق الأعمال، طبقة بنية تحتية تتواصل مع قاعدة البيانات وZoho وMoodle، وطبقة نماذج (domain) تعرّف أشكال البيانات المشتركة.

2 — Top-Level Folders

٢ — المجلدات الرئيسية

The repository root contains four significant directories:

moodle-zoho-integration-v3/
├── backend/          ← Python FastAPI server (this course's main focus)
├── moodle_plugin/    ← PHP plugin that gets installed on the Moodle server
├── course/           ← This course's HTML/CSS/JS files (what you're reading now)
└── .venv-2/          ← Python virtual environment (machine-local, not in git)
FolderLanguagePurpose

moodle_plugin/ PHP 8.x A standard Moodle local plugin that fires events (course enrolment, grade change, payment) and forwards them to the FastAPI backend via HTTP POST. It is installed on the Moodle server, not run locally.

They communicate exclusively over HTTP — the PHP plugin POSTs event data to the FastAPI backend's endpoints, and the backend responds with JSON.

يحتوي جذر المستودع على أربعة مجلدات رئيسية: backend/ (خادم Python FastAPI — محور هذه الدورة)، moodle_plugin/ (إضافة PHP تُثبَّت على خادم Moodle)، course/ (ملفات هذه الدورة التعليمية)، و.venv-2/ (البيئة الافتراضية المحلية غير المرفوعة إلى git).

فصل مهم: الخادم Python والإضافة PHP مستقلان تماماً. لا يشتركان في كود، ولا في عملية، ولا في قاعدة بيانات. يتواصلان حصراً عبر HTTP — الإضافة PHP ترسل بيانات الأحداث إلى نقاط نهاية FastAPI، والخادم يرد بـ JSON.

4 — backend/app/ — The Application Package

٤ — حزمة التطبيق backend/app/

Everything inside backend/app/ is a Python package. In Python, a package is simply a folder that contains a file named __init__.py. That file may be empty or may contain initialization code. Its presence tells Python "this directory is a package — you can import from it using dot notation (from app.core.config import settings)."

The top-level package structure inside app/:

app/
├── __init__.py
├── main.py              ← FastAPI application factory
├── api/
│   └── v1/
│       ├── router.py    ← aggregates all endpoint routers
│       └── endpoints/   ← one file per feature (students, grades, etc.)
├── core/
│   ├── config.py        ← reads .env into a Settings object
│   ├── scheduler.py     ← APScheduler background jobs
│   ├── security.py      ← HMAC verification
│   ├── idempotency.py   ← prevent duplicate webhook processing
│   └── logging.py       ← logging configuration
├── domain/
│   └── models/          ← Pydantic data models (schemas)
├── infra/
│   ├── db/              ← SQLAlchemy ORM models and session
│   ├── moodle/          ← Moodle REST API client
│   └── zoho/            ← Zoho CRM OAuth2 client
└── services/            ← business logic orchestration

app/main.py is the heart of the application. When uvicorn starts, it imports this file and looks for the object named app (the FastAPI instance). Here is what main.py does in order:

  1. Creates the FastAPI application: app = FastAPI(title=settings.APP_NAME)
  2. Adds CORS middleware — explained below.
  3. Defines a lifespan context manager that runs on startup (creates DB tables, starts the scheduler) and on shutdown (stops the scheduler cleanly).
  4. Mounts the API router at /api/v1: all REST endpoints live under this prefix.
  5. Mounts the admin router at /admin: serves the HTML admin panel.
  6. Adds a /health endpoint that returns {"status": "ok"}.

What is CORS? CORS stands for Cross-Origin Resource Sharing. When a browser page at https://moodle.school.com makes an HTTP request to http://backend.school.com:8001, the browser blocks the request by default because they have different origins (different domains or ports). CORS middleware tells the browser "this server explicitly allows requests from these origins" by adding special response headers. FastAPI's CORSMiddleware handles this automatically when you list the allowed origins.

كل شيء داخل backend/app/ هو حزمة Python. الحزمة هي مجلد يحتوي ملف __init__.py (قد يكون فارغاً) — وجود هذا الملف يخبر Python أن هذا المجلد حزمة يمكن الاستيراد منها بصيغة نقطية (from app.core.config import settings).

app/main.py هو قلب التطبيق. عند بدء uvicorn، يستورد هذا الملف ويبحث عن الكائن app (مثيل FastAPI). يقوم main.py بالترتيب بـ: إنشاء تطبيق FastAPI، إضافة CORS middleware، تعريف دورة حياة التطبيق (إنشاء جداول DB، تشغيل المجدوّل)، ربط راوتر API على /api/v1، وربط راوتر لوحة التحكم على /admin.

ما هو CORS؟ اختصار Cross-Origin Resource Sharing (مشاركة الموارد عبر الأصول المختلفة). عندما يحاول المتصفح إرسال طلب HTTP من moodle.school.com إلى backend:8001، يحجب المتصفح الطلب تلقائياً لاختلاف الأصول. CORS middleware يضيف ترويسات استجابة خاصة تخبر المتصفح أن هذا الخادم يسمح صراحةً بالطلبات من تلك الأصول.

5 — backend/app/api/v1/endpoints/ — The API Endpoints

٥ — نقاط نهاية API

An API endpoint is a URL that your server exposes and that external clients can call. For example, POST /api/v1/sync/students is an endpoint that triggers student synchronisation. Each endpoint is a Python function decorated with a FastAPI route decorator like @router.post("/sync/students").

Instead of putting all 50+ endpoints in a single 2000-line file, each feature area gets its own file. The v1/router.py then imports all these routers and combines them: 

# app/api/v1/router.py (simplified)
from fastapi import APIRouter
from .endpoints import sync_students, sync_grades, webhooks, mappings, ...

api_router = APIRouter()
api_router.include_router(sync_students.router, prefix="/sync", tags=["sync"])
api_router.include_router(sync_grades.router,   prefix="/sync", tags=["sync"])
api_router.include_router(webhooks.router,      prefix="/webhooks", tags=["webhooks"])
# ... and so on for every endpoint file

Key endpoint files and what they handle:

FileURL PrefixWhat it does
sync_students.py/api/v1/sync/studentsTriggers a pull of all student records from Moodle's core_user_get_users Web Service function and upserts them into the local students table.
sync_grades.py/api/v1/sync/gradesFetches grade data from Moodle for all enrolled students and syncs to the grades table.
sync_enrollments.py/api/v1/sync/enrollmentsSyncs course enrolment records — which student is enrolled in which course with which status.
sync_payments.py/api/v1/sync/paymentsPulls payment/order records from Moodle and stores them locally for reconciliation with Zoho CRM deals.
webhooks.py/api/v1/webhooks/zohoReceives incoming trigger notifications from Zoho CRM. Verifies HMAC signature, checks idempotency key, processes the payload, and returns 200 OK.
mappings.py/api/v1/mappingsCRUD endpoints for field mappings — the configuration table that defines which Zoho field name maps to which local database column.
zoho_push.py/api/v1/zoho/pushPushes locally stored student/enrolment/grade data up to Zoho CRM contacts and modules.
health.py/healthSimple health check endpoint. Returns database connectivity status. Called by monitoring systems.
debug.py/api/v1/debugDevelopment-only endpoints for inspecting internal state, testing API connections, and debugging sync issues. Disabled in production.
registrations.py/api/v1/registrationsManages student registration records — the bridge between a Zoho CRM lead/contact becoming a Moodle student.
programs.py/api/v1/programsProgramme management — maps Moodle course categories to Zoho programme records for structured enrolment tracking.

نقطة نهاية API هي URL يعرضه خادمك ويمكن للعملاء الخارجيين استدعاؤه. بدلاً من وضع جميع نقاط النهاية (50+) في ملف واحد ضخم، كل منطقة وظيفية لها ملفها الخاص. يقوم v1/router.py باستيراد جميع هذه الراوترات ودمجها. هذا يجعل الكود منظّماً: كل شخص يعرف أين يذهب حين يريد تعديل سلوك معين.

6 — backend/app/core/ — Cross-Cutting Concerns

٦ — المخاوف المشتركة app/core/

The core/ folder holds utilities that are needed everywhere in the application — not tied to any single feature, but used across all features. This pattern is called "cross-cutting concerns" in software architecture.

FileWhat it provides
config.py Defines a Settings class using pydantic_settings.BaseSettings. Pydantic reads all variables from the .env file and populates the class fields with type conversion and validation. A module-level singleton settings = Settings() is created once and imported everywhere: from app.core.config import settings. This means the entire app references config through a single strongly-typed object rather than scattered os.getenv() calls.
scheduler.py Sets up an APScheduler background scheduler that runs periodic sync jobs. Jobs are registered with their schedule (e.g., IntervalTrigger(minutes=15)) and the scheduler starts when the FastAPI application starts. Jobs run in a thread pool so they don't block the async event loop that handles HTTP requests.
security.py HMAC-SHA256 webhook signature verification. When Zoho sends a webhook POST, it includes an X-Zoho-Webhook-Token header that is an HMAC-SHA256 hash of the request body using ZOHO_WEBHOOK_HMAC_SECRET as the key. This file provides the verify_zoho_signature(body, header) function that recomputes the hash and compares it. Any request where they don't match is rejected with 401 Unauthorized.
idempotency.py Prevents duplicate processing of webhook events. Zoho may resend the same webhook if it doesn't receive a 200 OK quickly enough (network timeout). The idempotency system stores a fingerprint (hash of the webhook payload or a unique event ID) in the database. Before processing any webhook, it checks if that ID was already processed. If yes, the handler returns 200 OK immediately without re-running the business logic. This prevents double-writing to the database.
logging.py Configures Python's standard logging module for the whole application. Sets the log format (timestamp, level, module name, message), sets the log level from settings.LOG_LEVEL, and adds file handler if needed. Importing this module early in main.py ensures consistent logging everywhere.

مجلد core/ يحتوي الأدوات المساعدة المطلوبة في كل مكان بالتطبيق — غير مرتبطة بميزة واحدة بل تُستخدم عبر جميع الميزات.

config.py: يعرّف كلاس Settings يقرأ جميع متغيرات .env بالتحقق من الأنواع. يُنشأ كائن واحد مشترك (settings) يستورده الجميع. scheduler.py: يضبط APScheduler لتشغيل مزامنات دورية في الخلفية. security.py: يوفّر دالة التحقق من توقيع HMAC-SHA256 للـ webhooks الواردة. idempotency.py: يمنع معالجة نفس الـ webhook مرتين عبر تخزين بصمة الحدث. logging.py: يضبط نظام السجلّات (logs) للتطبيق بأكمله.

7 — backend/app/domain/ — Data Models

٧ — نماذج البيانات app/domain/

In software architecture, the domain layer defines the core data structures and business concepts of your application — independent of databases, HTTP, or external APIs. In this project, domain models are Pydantic classes that describe the shape of data flowing between layers: what a "student" looks like, what fields a "grade" has, etc.

Pydantic models provide automatic data validation and serialization. When FastAPI receives a JSON body and you declare the endpoint parameter as a Pydantic model, FastAPI automatically parses the JSON, validates each field type, raises a helpful 422 error if anything is wrong, and gives you a clean Python object.

Key domain models and what they represent:

طبقة النماذج (domain) تعرّف الهياكل البيانية الأساسية لتطبيقك — مستقلةً عن قواعد البيانات أو HTTP أو APIs الخارجية. في هذا المشروع، نماذج الـ domain هي كلاسات Pydantic تصف شكل البيانات المتدفقة بين الطبقات.

Pydantic توفّر التحقق التلقائي من البيانات والتسلسل (serialization). عندما تستقبل FastAPI جسم JSON وتعلن معامل النقطة النهائية بكلاس Pydantic، تقوم FastAPI تلقائياً بتحليل JSON والتحقق من كل نوع حقل ورفع خطأ 422 واضح إذا كان هناك خطأ.

8 — backend/app/infra/ — Infrastructure Layer

٨ — طبقة البنية التحتية app/infra/

The infrastructure layer is the boundary between your application code and external systems. Every dependency on something outside your Python process — the SQLite database, the Moodle web service API, the Zoho REST API — is isolated in this layer. The benefit: if you ever need to swap SQLite for PostgreSQL, you change code only in infra/db/ — not in business logic or API handlers.

infra/db/ — Database

  • base.py — Declares the SQLAlchemy Base class that all ORM models inherit from, and creates the engine from settings.DATABASE_URL.
  • session.py — Provides the get_db() FastAPI dependency. Each request gets its own database Session — a transaction context — which is automatically closed when the request ends. Usage: db: Session = Depends(get_db) in endpoint functions.
  • models.py — SQLAlchemy ORM table definitions — Python classes decorated with @mapper_registry.mapped that each correspond to a database table. These define the SQL schema: column names, types, indexes, foreign keys.

infra/moodle/ — Moodle API Client

  • users.py — Functions that call Moodle's Web Service REST API. For example, get_all_users() constructs a GET/POST request to {MOODLE_BASE_URL}/webservice/rest/server.php?wsfunction=core_user_get_users&moodlewsrestformat=json&wstoken={token} and returns the parsed user list.
  • All Moodle API calls go through this client — business logic never builds Moodle HTTP requests directly. This isolation means you can mock this client in tests.

infra/zoho/ — Zoho CRM Client

  • client.py — An httpx-based Zoho API client that handles OAuth2 token management automatically. It stores the current access token in memory. When a request fails with 401 (token expired), it automatically calls the Zoho token refresh endpoint using the ZOHO_REFRESH_TOKEN, gets a new access token, and retries the request. Your application code never deals with token refresh — the client handles it transparently.
  • Methods include: search_contacts(email), create_contact(data), update_contact(id, data), create_deal(data), get_module_fields(module).

طبقة البنية التحتية هي الحدود بين كود تطبيقك والأنظمة الخارجية. كل اعتماد على شيء خارج عملية Python — قاعدة البيانات SQLite، واجهة Moodle API، واجهة Zoho API — محصور في هذه الطبقة. الفائدة: لو أردت يوماً استبدال SQLite بـ PostgreSQL، تغيّر الكود في infra/db/ فقط دون تعديل منطق الأعمال أو معالجات API.

infra/db/: يحتوي تعريف قاعدة Python Base لـ SQLAlchemy، مزوّد الجلسة get_db()، وتعريفات جداول ORM. infra/moodle/: يحتوي عميل Moodle REST API. infra/zoho/client.py: عميل Zoho مع إدارة توكن OAuth2 التلقائية — عند انتهاء صلاحية التوكن، يجدّده العميل تلقائياً دون أن يهتم بذلك كود الأعمال.

9 — backend/app/services/ — The Service Layer

٩ — طبقة الخدمات app/services/

The service layer is where business logic lives. Services are Python functions (or classes) that orchestrate between the infrastructure layer and the API layer. An API handler should be thin — it validates input, calls a service function, and returns the result. The service function does the real work: calling Moodle, transforming data, calling Zoho, saving to the database.

Why separate services from endpoints? Because the same business logic may be needed in multiple places. The "sync students" logic needs to run both when a user clicks the button on the Admin Panel (POST /admin/sync) and when the scheduler runs it automatically every 15 minutes. By putting the logic in a service function, both can call the same function — no code duplication.

Key service functions:

ServiceWhat it orchestrates
sync_students_service(db)Calls infra.moodle.users.get_all_users(), transforms each user dict into a Student domain model, then upserts (insert or update) each record in the students table using SQLAlchemy. Returns a summary dict with counts of created/updated/skipped.
sync_grades_service(db)Fetches all enrolled students, calls Moodle grades API for each course, transforms results, upserts into grades table.
push_to_zoho_service(db)Reads students from the local DB that have changes since last push, calls infra.zoho.client.search_contacts(email) for each, then either creates a new Zoho contact or updates the existing one. Applies field mappings from the field_mappings table.
process_webhook_service(payload, db)Receives a verified Zoho webhook payload, identifies the event type (contact updated, deal created, etc.), maps the Zoho field changes to local DB columns using the field mapping table, then updates the relevant local records.
full_sync_service(db)Orchestrates a complete sync cycle: students → grades → enrolments → payments → push to Zoho. Called by the scheduler periodically and by the "Full Sync" button in the Admin Panel.

طبقة الخدمات هي حيث يعيش منطق الأعمال. الخدمات هي دوال Python تنسّق بين طبقة البنية التحتية وطبقة API. معالج API يجب أن يكون رقيقاً — يتحقق من المدخلات، يستدعي دالة خدمة، يُعيد النتيجة. دالة الخدمة تقوم بالعمل الحقيقي.

لماذا نفصل الخدمات عن نقاط النهاية؟ لأن نفس منطق الأعمال قد يُحتاج في أماكن متعددة. منطق "مزامنة الطلاب" يحتاج أن يعمل عند الضغط على الزر في لوحة التحكم وعند تشغيل المجدوّل تلقائياً. بوضع المنطق في دالة خدمة، يمكن لكلٍّ منهما استدعاء نفس الدالة — بدون تكرار كود.

10 — backend/admin/ — The Admin Panel

١٠ — لوحة التحكم backend/admin/

The admin panel is a full, server-rendered web UI accessible at http://localhost:8001/admin. It is built with FastAPI + Jinja2 templates — the server renders HTML on each request and sends complete HTML pages to the browser. This approach (server-side rendering) is simpler than building a single-page React app and does not require a separate frontend build step or API authentication token management.

admin/ folder structure:

admin/
├── router.py          ← ~1200 lines of route handlers (one per page)
├── auth.py            ← session-based login/logout logic
└── templates/         ← Jinja2 HTML files
    ├── base.html      ← shared sidebar/navbar layout

11 — backend/tests/ — Automated Tests

١١ — الاختبارات التلقائية backend/tests/

Testing means writing code that automatically verifies your application behaves correctly. Instead of manually clicking through the admin panel every time you make a change and hoping nothing broke, you run the test suite and it checks everything automatically in seconds.

This project uses pytest — the most popular Python testing framework. Test files are named test_*.py and each testing function starts with test_. Run all tests from inside backend/: 

pytest                          # run all tests
pytest -v                       # verbose: show each test name
pytest tests/test_sync.py       # run only one test file
pytest -k "student"             # run only tests with "student" in the name

Key test files:

FileWhat it tests
test_sync.pyTests the sync service functions with a mocked Moodle client. Verifies that student data is correctly upserted, that duplicates are handled, and that the sync event log is written.
test_webhooks.pyTests webhook endpoint with valid and invalid HMAC signatures. Verifies that tampered payloads are rejected with 401 and that valid payloads trigger the correct database updates.
test_zoho_client.pyTests the Zoho API client with mocked HTTP responses. Verifies token refresh logic, contact search, and update logic.
test_mappings.pyTests the field mapping CRUD endpoints.
test_admin_auth.pyTests the admin panel login flow: correct credentials create a session cookie, wrong credentials return a 401, and protected pages redirect to login without a valid session.
conftest.pypytest configuration: creates an in-memory SQLite test database and a FastAPI test client that all tests share. Fixtures (reusable setup/teardown) are defined here.

الاختبار يعني كتابة كود يتحقق تلقائياً أن تطبيقك يتصرف بشكل صحيح. بدلاً من النقر يدوياً عبر لوحة التحكم في كل مرة تجري تغييراً ومتمنياً ألا ينكسر شيء، تعدم مجموعة الاختبارات وتتحقق من كل شيء تلقائياً في ثوانٍ.

يستخدم المشروع pytest — إطار اختبار Python الأكثر شيوعاً. ملفات الاختبار تسمى test_*.py وكل دالة اختبار تبدأ بـ test_. شغّل جميع الاختبارات من داخل backend/: pytest أو بتفاصيل: pytest -v

12 — moodle_plugin/ — The Moodle PHP Plugin

١٢ — إضافة PHP لـ Moodle

The Moodle plugin is a standard Moodle local plugin written in PHP. It is installed on the Moodle server (not on the Python backend machine) and acts as the bridge that triggers events from Moodle activity into the FastAPI backend.

When something happens in Moodle — a student enrols, a grade is posted, a payment is processed — Moodle fires an internal event. The plugin listens for these events and, when one occurs, makes an HTTP POST request to the FastAPI backend's corresponding endpoint.

Key files in the plugin:

FilePurpose
version.php Required Moodle plugin manifest. Declares the plugin name (local_zohosync), version number, Moodle minimum version, and component string. Moodle reads this to register and track the plugin during installation and upgrade.
lib.php Standard plugin library file. May contain hook implementations and utility functions used by the plugin. Required to exist even if mostly empty.
settings.php Adds plugin configuration settings to the Moodle admin panel: the FastAPI backend URL and the shared webhook HMAC secret. Admins set these under Site Administration → Plugins → Local plugins → Zoho Sync.
classes/observer.php The event observer class. Contains a static method for each Moodle event type being monitored (e.g., user_enrolment_created, user_graded). Each method builds a JSON payload from the event data and POSTs it to the FastAPI endpoint via PHP's curl.
db/events.php A PHP array that registers which Moodle events the plugin listens to, and which observer class/method handles each one. Moodle reads this file at plugin installation time to wire up the event system.
ui/ Optional PHP UI pages that may appear under the Moodle admin panel for this plugin — for example, a connection test page that verifies the FastAPI backend is reachable from the Moodle server.

إضافة Moodle هي إضافة محلية Moodle قياسية مكتوبة بـ PHP. تُثبَّت على خادم Moodle (وليس على آلة Python) وتعمل كحلقة وصل تُطلق الأحداث من نشاط Moodle إلى FastAPI.

عندما يحدث شيء في Moodle — تسجيل طالب، إضافة درجة، معالجة دفعة — يُطلق Moodle حدثاً داخلياً. الإضافة تستمع لهذه الأحداث، وعند حدوثها، ترسل POST HTTP إلى نقطة النهاية المقابلة في FastAPI.

الملفات المهمة: version.php (ملف بيانات الإضافة المطلوب من Moodle)، settings.php (إعدادات في لوحة Moodle: رابط FastAPI والـ HMAC secret)، classes/observer.php (الكلاس الذي يستقبل الأحداث ويرسل POST إلى FastAPI)، db/events.php (يسجّل الأحداث التي تستمع لها الإضافة).
