1. What is Moodle?

١. ما هو Moodle؟

Moodle is an open-source Learning Management System (LMS) built with PHP. Colleges use it to deliver online courses: students enroll, watch materials, submit assignments, and receive grades — all inside one platform. Our integration keeps student account data in Moodle always in sync with Zoho CRM.

Moodle هو نظام إدارة التعلم (LMS) مفتوح المصدر مبني بالـ PHP. يستخدمه المعهد لتقديم الكورسات الإلكترونية: يسجّل الطلاب في الكورسات، يشاهدون المحتوى، يسلّمون الواجبات، ويستلمون الدرجات — كلها داخل منصة واحدة. تكاملنا يجعل بيانات حسابات الطلاب في Moodle دائماً متزامنة مع Zoho CRM. حين يُضاف طالب جديد في Zoho يُنشأ حسابه تلقائياً في Moodle، وحين تتغير درجاته في Moodle تُحدَّث في Zoho.

2. What Does Our Plugin Do?

٢. ماذا تفعل إضافتنا؟

The plugin (local_moodle_zoho_sync) has three jobs:

  1. Event listener: Catches Moodle events (enrollment, grade update, user creation) and sends a webhook POST to the FastAPI backend so Zoho stays in sync.
  2. Custom Web Service functions: API endpoints the backend calls to create Moodle users, enroll them, and fetch grades.
  3. Student Dashboard UI: A 4-tab page embedded in Moodle showing each student their programme, classes, grades, and student ID card — sourced from local SQLite.

الإضافة (local_moodle_zoho_sync) لها ثلاث مهام رئيسية: (١) مستمع الأحداث: تلتقط أحداث Moodle (تسجيل، تحديث درجة، إنشاء مستخدم) وترسل webhook إلى الباك-إند FastAPI. (٢) دوال Web Service مخصصة: يستخدمها الباك-إند لإنشاء مستخدمي Moodle وتسجيلهم وجلب درجاتهم. (٣) واجهة لوحة الطالب: صفحة بـ 4 تبويبات مدمجة في Moodle تعرض لكل طالب برنامجه وفصوله ودرجاته وبطاقته الجامعية — مصدرها SQLite المحلية لضمان السرعة.

3. File-by-File Walkthrough

٣. شرح كل ملف

version.php — Plugin Identity

The first file Moodle reads. Declares the component name (must match folder name exactly), version number, and minimum Moodle version required.

أول ملف يقرأه Moodle عند فحص الإضافات. يُعرِّف اسم المكوّن — يجب أن يطابق اسم المجلد تماماً وإلا يرفض Moodle الإضافة. رقم الإصدار بصيغة YYYYMMDDNN.

<?php
defined('MOODLE_INTERNAL') || die();
$plugin->component = 'local_moodle_zoho_sync';  // MUST match folder name
$plugin->version   = 2024010100;  // YYYYMMDDNN
$plugin->requires  = 2022112800;  // Moodle 4.1 minimum
$plugin->maturity  = MATURITY_STABLE;

db/events.php — Declare Events to Observe

Tells Moodle which events to listen for and which class method to call when each fires. Without this file, no events are captured.

يخبر Moodle بالأحداث التي يجب الاستماع إليها وأي دالة تُستدعى عند كل حدث. بدونه لن يلتقط النظام أي حدث إطلاقاً. الاسم الكامل للحدث مع نطاق PHP ضروري (مثلاً \core\event\user_enrolment_created).

<?php
defined('MOODLE_INTERNAL') || die();
$observers = [
    [
        'eventname'  => '\core\event\user_enrolment_created',
        'callback'   => 'local_moodle_zoho_sync\observer::enrolment_created',
    ],
    [
        'eventname'  => '\core\event\user_graded',
        'callback'   => 'local_moodle_zoho_sync\observer::user_graded',
    ],
    [
        'eventname'  => '\core\event\user_created',
        'callback'   => 'local_moodle_zoho_sync\observer::user_created',
    ],
    [
        'eventname'  => '\core\event\user_updated',
        'callback'   => 'local_moodle_zoho_sync\observer::user_updated',
    ],
    [
        'eventname'  => '\core\event\user_enrolment_deleted',
        'callback'   => 'local_moodle_zoho_sync\observer::enrolment_deleted',
    ],
];

classes/observer.php — The Event Handler

Each method runs when its event fires. Builds a JSON payload and sends it as an HMAC-signed POST to the FastAPI backend. The signature verifies the request really came from Moodle.

كل دالة تُشغَّل حين يحدث حدثها المسجَّل. تبني حمولة JSON وترسلها كطلب HTTP POST موقَّع بـ HMAC إلى الباك-إند. التوقيع يضمن أن الباك-إند يقبل فقط الطلبات القادمة من Moodle الفعلي — أي طلب بتوقيع خاطئ يُرفض تلقائياً.

<?php
namespace local_moodle_zoho_sync;
class observer {
    public static function enrolment_created(\core\event\user_enrolment_created $event) {
        $data    = $event->get_data();
        $payload = json_encode([
            'event'     => 'enrollment_created',
            'user_id'   => $data['relateduserid'],
            'course_id' => $data['courseid'],
            'time'      => $data['timecreated'],
        ]);
        $webhook_url = get_config('local_moodle_zoho_sync', 'webhook_url');
        $secret      = get_config('local_moodle_zoho_sync', 'webhook_secret');
        $signature   = hash_hmac('sha256', $payload, $secret);
        $curl = new \curl();
        $curl->setHeader(['Content-Type: application/json',
                          'X-Moodle-Signature: ' . $signature]);
        $curl->post($webhook_url . '/api/v1/moodle/events', $payload);
    }
}

settings.php — Admin Config Page

Adds a settings page under Site Admin → Plugins → Local plugins → Moodle-Zoho Integration where you enter the backend URL and webhook secret.

يُضيف صفحة إعدادات تحت Site Admin → Plugins → Local plugins. هذه الصفحة تلصق فيها رابط الباك-إند وكلمة سر الـ webhook المشتركة. يجب أن تتطابق مع قيمة WEBHOOK_SECRET في backend/.env تماماً وإلا سيرفض الباك-إند كل الطلبات.

<?php
defined('MOODLE_INTERNAL') || die();
if ($ADMIN->fulltree) {
    $settings->add(new admin_setting_configtext(
        'local_moodle_zoho_sync/webhook_url',
        'Backend Webhook URL',
        'Full URL of your FastAPI backend',
        'http://localhost:8001', PARAM_URL
    ));
    $settings->add(new admin_setting_configtext(
        'local_moodle_zoho_sync/webhook_secret',
        'Webhook Secret',
        'Must match WEBHOOK_SECRET in backend .env exactly',
        '', PARAM_TEXT
    ));
}

4. Installing the Plugin — Step by Step

٤. تثبيت الإضافة خطوة بخطوة

Step 1 — Copy the plugin folder

Copy moodle_plugin/ to /var/www/moodle/local/moodle_zoho_sync/ on the Moodle server. The folder name must be exactly moodle_zoho_sync.

انسخ مجلد moodle_plugin/ من المشروع إلى /var/www/moodle/local/moodle_zoho_sync/ على سيرفر Moodle. اسم المجلد يجب أن يكون moodle_zoho_sync تماماً — يجب أن يطابق $plugin->component في version.php أو سيرفض Moodle الإضافة ولن تظهر في القائمة.

Step 2 — Run the Moodle upgrade

Log in as Moodle admin → Site Administration → Notifications. Moodle scans for new plugins automatically. Click Upgrade Moodle database now.

سجّل دخولك كمدير → Site Administration → Notifications. يفحص Moodle الإضافات الجديدة تلقائياً ويعرض شاشة ترقية. اضغط Upgrade Moodle database now. إذا ظهر مربع لقبول الشروط، وافق عليه وأكمل العملية.

Step 3 — Configure plugin settings

Site Admin → Plugins → Local plugins → Moodle-Zoho Integration. Enter the Backend Webhook URL (http://localhost:8001 for local dev) and Webhook Secret (must match WEBHOOK_SECRET in backend/.env).

اذهب إلى Site Admin → Plugins → Local plugins → Moodle-Zoho Integration. أدخل رابط الباك-إند وكلمة سر الـ webhook. يجب أن تطابق WEBHOOK_SECRET في backend/.env تماماً. إذا اختلفا سيرفض الباك-إند كل الطلبات الواردة من Moodle بخطأ 401 Unauthorized.

Local dev note: If Moodle is on a different machine it cannot reach localhost:8001. Use ngrok: run ngrok http 8001, copy the https URL and use it as the Backend URL.

ملاحظة التطوير المحلي: إذا كان Moodle على جهاز مختلف لن يستطيع الوصول إلى localhost:8001. استخدم ngrok: شغّل ngrok http 8001، انسخ الرابط https الذي يعطيك، والصقه كـ Backend URL.

5. Moodle Web Services API Token Setup

٥. إعداد رمز API لخدمات الويب في Moodle

Moodle exposes a REST API for external systems to call. Steps: enable web services, enable REST protocol, create External Service, add functions, generate a token.

Moodle يكشف REST API للأنظمة الخارجية. للاستخدام يجب: تفعيل خدمات الويب، تفعيل بروتوكول REST، إنشاء External Service، إضافة دوال إليه، ثم توليد token لمستخدم مدير. كل خطوة ضرورية — تخطّي أي منها يمنع الاتصال تماماً.

Step 1 — Enable Web Services

Site Administration → Advanced features → tick Enable web services → Save changes

Site Administration → Advanced features → فعّل Enable web services → Save changes. هذا المفتاح الرئيسي — بدونه لن يعمل أي token.

Step 2 — Enable REST Protocol

Site Admin → Plugins → Web services → Manage protocols → Enable REST

Site Admin → Plugins → Web services → Manage protocols → فعّل بروتوكول REST. يجب تفعيله صراحةً حتى يُقبَل moodlewsrestformat=json في الطلبات.

Step 3 — Create External Service

Site Admin → Web services → External services → Add → name it "Zoho Sync Backend", tick Enabled and Authorised users only → Save

Site Admin → Web services → External services → Add → سمّيه "Zoho Sync Backend"، فعّل Enabled وAuthorised users only → Save. خيار Authorised users only يعني يعمل الرمز فقط مع المستخدمين المرتبطين صراحةً بالخدمة.

Step 4 — Add Functions to the Service

Click Add functions and add: core_user_create_users, core_user_get_users, core_user_update_users, enrol_manual_enrol_users, enrol_manual_unenrol_users, core_grades_get_grades, gradereport_user_get_grade_items

اضغط Add functions وأضف هذه الدوال. كل دالة تمثّل عملية: إنشاء مستخدم، جلب مستخدم، تحديث مستخدم، تسجيل طالب، إلغاء تسجيل، جلب الدرجات. بدون إضافة الدالة هنا سيرفض Moodle استدعاءها حتى لو كان الـ token صحيحاً.

Step 5 — Generate API Token

Site Admin → Web services → Manage tokens → Add → select a Moodle admin user → select "Zoho Sync Backend" → Save. Copy the token and paste it as MOODLE_TOKEN in backend/.env.

Site Admin → Web services → Manage tokens → Add → اختر مستخدم مدير → اختر "Zoho Sync Backend" → Save. انسخ الـ token والصقه كـ MOODLE_TOKEN في backend/.env. هذا الرمز هو ما يستخدمه باك-إندنا للتحقق من هويته في كل طلب API إلى Moodle.

🖥 UI Test: Verify Token Works
  1. Open http://localhost:8001/docs
  2. Find endpoint GET /api/v1/moodle/test-connection
  3. Click Try it out → Execute
  4. Expected: {"status": "ok", "moodle_version": "..."}
  5. If error: check MOODLE_BASE_URL and MOODLE_TOKEN in .env

6. How the Backend Calls the Moodle API

٦. كيف يستدعي الباك-إند Moodle API

All Moodle REST calls follow one pattern: POST to /webservice/rest/server.php with wstoken, wsfunction, and function parameters as form fields. Response is always JSON.

جميع استدعاءات Moodle REST تتبع نمطاً واحداً: POST إلى /webservice/rest/server.php مع الـ token واسم الدالة وبارامتراتها كحقول form. الاستجابة دائماً JSON. أهم تحذير: Moodle يعطي HTTP 200 حتى عند الخطأ — الأخطاء تأتي كحقل exception في JSON بدلاً من HTTP 400 أو 500.

import httpx
from app.core.config import settings

class MoodleClient:
    def __init__(self):
        self.base  = f"{settings.MOODLE_BASE_URL}/webservice/rest/server.php"
        self.token = settings.MOODLE_TOKEN

    def call(self, function: str, params: dict = {}) -> dict:
        payload = {"wstoken": self.token,
                   "wsfunction": function,
                   "moodlewsrestformat": "json",
                   **params}
        resp = httpx.post(self.base, data=payload)
        resp.raise_for_status()
        result = resp.json()
        # CRITICAL: Moodle returns HTTP 200 even on failure!
        if isinstance(result, dict) and "exception" in result:
            raise Exception(f"Moodle error: {result.get('message', result)}")
        return result

    def create_user(self, first_name, last_name, email, username):
        return self.call("core_user_create_users", {
            "users[0][firstname]": first_name,
            "users[0][lastname]":  last_name,
            "users[0][email]":     email,
            "users[0][username]":  username,
            "users[0][password]":  "TempPass1!",
            "users[0][auth]":      "manual",
        })

    def enrol_student(self, user_id: int, course_id: int):
        return self.call("enrol_manual_enrol_users", {
            "enrolments[0][roleid]":   5,
            "enrolments[0][userid]":   user_id,
            "enrolments[0][courseid]": course_id,
        })

moodle_client = MoodleClient()
Critical: Moodle always returns HTTP 200 even on errors. Always check for the "exception" key. The call() method above handles this automatically.

تحذير حرج: Moodle يعطي HTTP 200 دائماً حتى عند الخطأ. يجب دائماً فحص حقل "exception" في كل استجابة. دالة call() تفعل هذا تلقائياً — لكن إن أرسلت طلباً خاماً يجب إضافة هذا الفحص يدوياً.

إدارة الربط بين رموز البرامج/الوحدات BTEC (من Zoho) ومعرّفات الكورسات في Moodle. عند إنشاء تسجيل جديد يُراجع النظام هذه القوالب. إذا لم يجد تطابقاً لا يُسجَّل الطالب في أي كورس — لذا يجب الحرص على اكتمالها.

8. Student Dashboard UI Pages — Complete Guide (7 Pages)

٨. صفحات لوحة الطالب — دليل كامل (٧ صفحات)

When a logged-in student opens their Moodle profile, the plugin injects a 4-tab dashboard below the standard profile info. Data is served from local SQLite (not live Zoho calls) — fast and always available. All files in moodle_plugin/ui/student/.

عندما يفتح الطالب ملفه الشخصي في Moodle، تحقن إضافتنا لوحة بـ 4 تبويبات أسفل معلومات Moodle القياسية. البيانات من SQLite المحلية وليس من Zoho مباشرةً — مما يجعل اللوحة سريعة ومتاحة حتى لو كانت Zoho بطيئة. جميع الصفحات في moodle_plugin/ui/student/.

Student Tab 1

profile.php — My Profile

ملفي الشخصي

Displays student's current data: name, email, student ID, department, branch, and profile photo. Edit button opens a form to submit a change request to BTEC_Student_Requests — must be admin-approved before applied.

يعرض بيانات الطالب: الاسم، الإيميل، رقم الطالب، القسم، الفرع، والصورة. زر تعديل يفتح نموذج طلب تغيير يُرسَل إلى BTEC_Student_Requests في Zoho ويحتاج موافقة المدير. يضمن ألا يغيّر طالب بياناته الرسمية دون مراجعة.

Student Tab 2

programs.php — My Programmes

برامجي الأكاديمية

Programme registration details from BTEC_Registrations: programme name, level, start date, expected end date, registration status (Active / Completed / Suspended), and payment status.

تفاصيل تسجيل الطالب من BTEC_Registrations: اسم البرنامج، المستوى، تاريخ البدء، تاريخ الانتهاء المتوقع، حالة التسجيل (نشط/مكتمل/معلّق)، وحالة الدفع. يُمكّن الطالب من رؤية وضعه الأكاديمي دون التواصل مع الإدارة.

Student Tab 3

classes.php — My Classes

فصولي الدراسية

Class schedule from BTEC_Classes and BTEC_Enrollments: class name, dates, instructor, room/online link, enrollment status. Links to the corresponding Moodle course.

جدول الفصول من BTEC_Classes وBTEC_Enrollments: اسم الفصل، التواريخ، المحاضر، القاعة/الرابط الإلكتروني، وحالة التسجيل. روابط مباشرة للكورس المقابل في Moodle. كل ما هو مسجَّل فيه في مكان واحد.

Student Tab 4

student_card.php — My Student Card

بطاقتي الجامعية

Digital student ID card: profile photo (via serve_photo.php), full name, student ID, programme, department, academic year, and auto-generated QR code. Print button opens card_print.php.

بطاقة هوية رقمية: الصورة (عبر serve_photo.php)، الاسم الكامل، رقم الطالب، البرنامج، القسم، السنة الأكاديمية، ورمز QR يُولَّد تلقائياً. زر طباعة يفتح card_print.php. يستطيع الطالب طباعتها كبطاقة مؤقتة.

Student Page 5

requests.php — My Change Requests

طلبات التغيير

History of all change requests submitted via profile.php: what was requested, status (Pending / Approved / Rejected), date submitted, admin notes if rejected. Students can cancel pending requests.

سجل كل طلبات التغيير: ما طُلب تغييره، الحالة (قيد الانتظار / مقبول / مرفوض)، تاريخ الإرسال، وملاحظات المدير عند الرفض. يستطيع الطالب إلغاء الطلبات المعلّقة إذا غيّر رأيه.

Helper

serve_photo.php — Photo Server

خادم الصور

Secure endpoint serving the student's profile photo. Students can only see their own photo; admins see all. Called internally by profile.php and student_card.php via <img> tag.

نقطة نهاية آمنة تُقدّم صورة الطالب. الطالب يرى صورته الخاصة فقط؛ المدير يرى الكل. يُستدعى داخلياً من profile.php و student_card.php. لا تفتحه مباشرةً إلا لأغراض التصحيح.

Helper

card_print.php — Print Card

طباعة البطاقة

Print-optimised version of student_card.php: no Moodle header/footer, white background, A4 sizing. Opened automatically when the student clicks Print.

نسخة محسَّنة للطباعة: بلا header/footer لـ Moodle، خلفية بيضاء، مقاسات A4 صحيحة. تُفتح تلقائياً عند الضغط على زر الطباعة في student_card.php.
